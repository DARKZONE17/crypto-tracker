// Chapter 1 — Why Does Git Exist?

The Problem Git Solves

Before learning any commands, you need to understand the pain that Git was built to fix. If you've ever done any of the following things, you've already felt the problem.

THE REPORT.DOC NIGHTMARE — WHAT DEVELOPERS DID BEFORE GIT

Imagine you're writing a project and managing it like this:

project_folder/ ├── report_v1.doc ├── report_v2.doc ├── report_final.doc ├── report_final_FINAL.doc ├── report_final_real.doc └── report_DONT_DELETE_THIS_ONE.doc ← 😅 we've all been here

Now ask yourself these questions — and notice you can't answer any of them confidently:

  • What actually changed between v2 and final?
  • If final_real.doc is broken, how do you get the last working version back?
  • If two people edited different versions — how do you combine them?
  • Which is the "real" current version?

This is exactly the chaos Git eliminates. Instead of messy file copies, Git tracks every change automatically, with timestamps, author names, and descriptions. One folder. Full history. No confusion.

WHAT GIT IS — THE ACTUAL DEFINITION

Git is a Version Control System (VCS) — software that tracks changes in your code over time.

Every time you tell Git to save a snapshot (called a commit), it records the exact state of every file at that moment. You can go back to any past snapshot instantly. You can work on experimental features without touching your working code. Multiple people can work on the same project without overwriting each other's work.

🧠 Mental Model — Remember This Git is simultaneously three things:

🧠 A time machine — go back to any point in your project's history
📸 A camera — takes snapshots (commits) of your code at moments you choose
🌳 A multiverse — lets you create parallel universes (branches) of your code
☁️ A cloud backup system — when paired with GitHub, your code is safe even if your laptop dies
GIT vs GITHUB — THEY ARE NOT THE SAME THING

Git

  • Software installed on your laptop
  • Works completely offline — no internet needed
  • A command-line tool you type commands into
  • Free and open source, created by Linus Torvalds (creator of Linux) in 2005
  • Tracks your code history locally

GitHub

  • A website that stores Git repos in the cloud
  • Requires internet and an account
  • Adds a visual interface, collaboration features, PRs
  • Owned by Microsoft
  • Alternatives: GitLab, Bitbucket, Azure DevOps
Simple Way to Remember Git = the tool (runs on your laptop). GitHub = cloud storage for that tool (runs on the internet). You can use Git without GitHub. You cannot use GitHub without Git.

Key Concepts — Understand These Deeply

Most beginners jump straight to commands and get confused. Don't do that. These five concepts are the mental model behind everything. If you understand these, every command will make intuitive sense.

CONCEPT 1 — REPOSITORY (REPO)

A repository is simply your project folder — tracked by Git. When you run git init inside a folder, Git creates a hidden .git subfolder. That hidden folder IS Git's brain — it stores every commit, every branch, the complete history of every file. Never delete the .git folder or you lose all history.

Terminal
mkdir crypto-tracker && cd crypto-tracker git init Initialized empty Git repository in /crypto-tracker/.git/ ls -la drwxr-xr-x .git ← this is Git's brain — don't touch it
CONCEPT 2 — COMMIT (THE MOST IMPORTANT CONCEPT)

A commit is a saved snapshot of your entire project at a specific moment in time. Every commit has a unique ID (a 40-character hash like a3f5d9c...), a timestamp, the author's name, and a message you write describing what changed.

Think of commits like save points in a video game. You can always go back to any save point. The difference from a regular "save" is that commits are permanent and stackable — you can have thousands of them and jump to any one.

📸 Analogy — Photography A commit is like a photograph of your code. The photo is frozen in time — it never changes. You can take as many photos as you want (commit often!). You can always look at an old photo to see how things looked before. The photos are stored in your repo's history in order.
CONCEPT 3 — STAGING AREA (THE MIDDLE STEP)

This confuses every beginner. There are three zones in Git, not two:

Working Directory
files on disk
→
Staging Area
selected changes
→
Local Repo
saved history
→
Remote (GitHub)
cloud backup

Working Directory — where you actually edit files. Changes here are "unsaved" from Git's perspective.

Staging Area — a holding area. You use git add to move changes here. Think of it as selecting which changes you want to include in your next commit. This is why staging exists: if you changed 5 files but only want to commit 2 of them, you add only those 2 to staging.

Local Repo — the saved history on your laptop. When you run git commit, everything in staging becomes a permanent commit here.

🛒 Analogy — Supermarket Shopping Working = items on shelves (you can look at everything). Staging = your shopping cart (you've decided to buy these). Commit = paying at checkout (the purchase is finalized, permanent). You pick what goes in your cart before you pay — that's staging.
CONCEPT 4 — BRANCH (PARALLEL UNIVERSES)

A branch is an independent line of development. It's like creating a copy of your project where you can make changes freely without affecting the original. When you're happy with the changes, you merge them back.

Every Git repo starts with one branch called main (or master in older repos). This is your "official" code — always working, always stable. When you want to add a new feature or fix a bug, you create a new branch, do your work there, then merge it back into main.

🌳 Analogy — Road Construction main branch = the main highway (always open, always working).
feature branch = a detour road being built in parallel.
When the detour is ready, it becomes part of the main highway (merge).
Traffic on the main highway was never disrupted during construction.
CONCEPT 5 — REMOTE (CLOUD BACKUP)

A remote is a copy of your repo stored somewhere else — usually GitHub. When you push to a remote, your commits are uploaded to the cloud. This serves two purposes: backup (laptop crashes → code is safe) and collaboration (teammates can download your commits).

The most common remote name is origin — that's just the conventional name for the main remote. You can have multiple remotes with different names, but 99% of the time you'll only use origin.

// Chapter 3 — How Git Is Structured

Git Architecture — The Four Zones

This is the most important diagram to memorize. Every Git command moves data between these four zones. Once you know this, you always know what a command is doing.

───────────────────────────────────────────────────────────────────────────── Working Directory → Staging Area → Local Repo → Remote (GitHub) ───────────────────────────────────────────────────────────────────────────── Files you edit Files you Committed Cloud copy on your disk selected with history on on GitHub (unsaved to Git) git add your laptop (pushed) ───────────────────────────────────────────────────────────────────────────── git add ──────► git commit ─────► git push ──────► ◄─────────────────────────────────────────────── git pull ◄────────────── git fetch ◄───────────────── git checkout ─────────────────────────────────────────────────────────────────────────────
HOW TO USE THIS DIAGRAM Every time you run a Git command and feel confused about what it's doing — come back to this diagram. git add moves changes from Working → Staging. git commit moves from Staging → Local Repo. git push moves from Local Repo → Remote. git pull moves from Remote → Working Directory (it's fetch + merge). That's all Git is — moving snapshots of code between these four zones.
// Chapter 4 — Day One Skills

Basic Workflow — Your First Git Session

These are the commands you will type every single day. Learn this loop until it's automatic. We'll go through it step by step with full explanations of what each command actually does.

1
Initialize a repo — tell Git to start tracking this folder
Run this once, inside your project folder. Creates the hidden .git folder. You never need to run this again for the same project.
Terminal
cd my-project # navigate into your project folder first git init Initialized empty Git repository in /my-project/.git/
2
Check status — what has changed since the last commit?
Run this constantly — before adding, before committing, whenever you feel confused. It tells you exactly what state your files are in across all three zones.
Terminal
git status On branch main Changes not staged for commit: ← in Working Dir, not staged yet modified: producer.py Untracked files: ← brand new file, Git has never seen it consumer.py # Red = not staged. Green = staged. This command is your best friend.
3
Add files — move changes to the staging area
This does NOT save anything yet. It just says "I want to include these changes in the next commit." You're filling the shopping cart before checkout.
Terminal
git add producer.py # stage one specific file git add producer.py consumer.py # stage multiple specific files git add . # stage ALL changed/new files in current folder git status Changes to be committed: ← these are now staged (green) new file: consumer.py modified: producer.py
IMPORTANT git add . adds everything including files you may not want tracked (like .env secrets, __pycache__, etc.). Always create a .gitignore file first to exclude those. See the cheatsheet section for a ready-made .gitignore.
4
Commit — save the snapshot permanently
This creates a permanent, timestamped record of everything in the staging area. The -m flag lets you write the message inline. Write messages that explain what changed and why — future you will be grateful.
Terminal
git commit -m "feat: add Kafka producer and consumer" [main a3f5d9c] feat: add Kafka producer and consumer 2 files changed, 87 insertions(+) # a3f5d9c is the unique commit hash — Git's ID for this snapshot

❌ Bad commit messages

git commit -m "stuff"
git commit -m "fix"
git commit -m "asdf"
git commit -m "changes"

✅ Good commit messages

feat: add Kafka producer
fix: resolve offset reset error
docs: update architecture diagram
refactor: extract config to utils
5
View history — see all past commits
Shows every commit, newest first. Each entry shows the hash, author, date, and message.
Terminal
git log commit a3f5d9c7b1e2... (HEAD → main) Author: Your Name <you@email.com> Date: Thu Apr 25 10:23:01 2024 feat: add Kafka producer and consumer git log --oneline # compact version — much easier to read a3f5d9c feat: add Kafka producer and consumer e9b2c41 docs: add README f1d3a87 Initial commit
// Chapter 5 — Inspecting Changes

Understanding Changes — diff

Before you commit, you want to review exactly what changed. git diff shows you the exact lines that were added or removed.

git diff — WHAT CHANGED (NOT YET STAGED)

Shows changes in your Working Directory that you have NOT yet staged with git add.

Terminal
git diff diff --git a/producer.py b/producer.py --- a/producer.py +++ b/producer.py @@ -10,7 +10,7 @@ - TOPIC = "hello-topic" ← removed line (old) + TOPIC = "crypto-prices" ← added line (new) # Red lines (starting with -) = what was removed # Green lines (starting with +) = what was added
git diff --staged — WHAT'S IN THE STAGING AREA

Shows changes that you HAVE staged with git add — what will go into the next commit.

Terminal
git add producer.py git diff --staged # shows what's staged (what the next commit will contain) # git diff alone would now show nothing (you staged everything)
WORKFLOW TIP Before every commit: run git diff --staged to review exactly what you're about to commit. This prevents accidentally committing debug code, API keys, or half-finished changes.
// Chapter 6 — The Most Powerful Git Feature

Branching — Working Safely in Parallel

Branching is the feature that makes Git truly powerful. Without branches, you'd always be editing the same code — every change risks breaking what already works. With branches, you can experiment freely in isolation and only merge back when things work.

WHY BRANCHES? THE REAL REASON

Imagine you have a working crypto tracker. You want to add WebSocket support. Without branches, you start editing the working producer.py directly. Halfway through, it's broken. Now your whole project is broken — there's no "working version" to fall back to easily.

With branches: you create feature/websocket, do all your experimental WebSocket work there. Your main branch still has the working producer.py, completely untouched. If the feature works → merge it in. If it's a disaster → delete the branch, main is perfectly fine.

main: ●─────────────────────────────────────────────────────● ← always stable │ ↑ │ git checkout -b feature/websocket merge │ ↓ │ feature: ●──●──●──●──● ← you work here, safely isolated, main untouched │ │ │ │ └── fix: handle disconnect and reconnect │ │ │ └───── feat: parse Binance tick data format │ │ └──────── feat: connect to wss://stream.binance.com │ └─────────── chore: pip install websockets └────────────── branch created from main
ALL BRANCH COMMANDS
Terminal
# ── See all branches ────────────────────────────────────── git branch * main ← the asterisk shows your current branch feature/websocket git branch -a # -a shows remote branches too * main remotes/origin/main remotes/origin/feature/websocket # ── Create a new branch ─────────────────────────────────── git branch feature/websocket # creates branch but stays on current branch # ── Switch to a branch ──────────────────────────────────── git checkout feature/websocket # classic way to switch git switch feature/websocket # modern alternative (Git 2.23+) # ── Create AND switch in one command (use this most often) ─ git checkout -b feature/websocket Switched to a new branch 'feature/websocket' # ── Merge branch back into main ─────────────────────────── git checkout main # switch to main first git merge feature/websocket # merge feature into main Updating a3f5d9c..b1e2c3d Fast-forward ws_producer.py | 87 +++++++++++++++ # ── Delete a branch (after merging — clean up) ──────────── git branch -d feature/websocket # -d is safe: refuses if unmerged git branch -D feature/websocket # -D forces delete even if unmerged
BRANCH NAMING CONVENTION Use type/description format. Types: feature/ (new functionality), fix/ (bug fixes), docs/ (documentation), refactor/ (code restructuring), chore/ (maintenance).
Examples: feature/kafka-producer · fix/consumer-offset-bug · docs/update-readme
// Chapter 7 — The Reality of Collaborative Development

Merge Conflicts — What They Are and How to Fix Them

Merge conflicts sound scary but they're completely normal. Every developer hits them. They happen when two branches both changed the same line of the same file — and Git can't decide which version to keep. Your job is to look at both versions and decide.

WHEN DO CONFLICTS HAPPEN?

You edited line 10 of producer.py on your feature branch. Your teammate also edited line 10 of producer.py on their branch. Both branches were merged into main. Git sees two different versions of line 10 — it doesn't know which one is "right". So it stops and asks you to decide. That's a conflict.

Terminal
git merge feature/websocket Auto-merging producer.py CONFLICT (content): Merge conflict in producer.py Automatic merge failed; fix conflicts and then commit the result. # Git didn't crash — it's asking you to resolve the conflict manually
WHAT A CONFLICTED FILE LOOKS LIKE

Git modifies the conflicted file and adds conflict markers. Open the file and you'll see:

# producer.py — Git has added conflict markers to this file

<<<<<<< HEAD
TOPIC = "crypto-prices"      ← YOUR version (the current branch you're on)
=======
TOPIC = "crypto-prices-dev"  ← INCOMING version (the branch being merged in)
>>>>>>> feature/websocket

# Everything between <<<<<<< HEAD and ======= is your version
# Everything between ======= and >>>>>>> is the incoming version

To fix it: edit the file manually. Delete the conflict markers and keep whichever version is correct (or combine them). The file should look completely normal when you're done — no markers left.

Terminal — after fixing the conflict in the file
# After editing the file to resolve the conflict: git add producer.py # tell Git: "I've resolved the conflict in this file" git commit -m "merge: resolve TOPIC conflict — use dev topic name" [main d4e5f6g] merge: resolve TOPIC conflict
VS CODE TIP — EASIEST WAY TO RESOLVE CONFLICTS Open the conflicted file in VS Code. You'll see colour-coded panels and clickable buttons: "Accept Current Change" (keep yours), "Accept Incoming Change" (keep theirs), "Accept Both Changes" (keep both, one after the other). Click the right one and the markers disappear automatically.

A remote is a copy of your repo on a server. GitHub is the most popular hosting service. Connecting your local repo to a remote lets you back up your code and collaborate with others.

CONNECTING AND USING A REMOTE

PULL REQUEST (PR) — THE PROFESSIONAL WAY TO MERGE

In real teams you don't merge branches locally — you open a Pull Request on GitHub. A PR shows a visual diff of all your changes and allows teammates to review and comment before the merge happens. This is called code review.

Push your feature branch to GitHub

2
Go to GitHub — click "Compare & pull request"

// Chapter 9 — Classic Interview Question

git pull vs git fetch — The Difference

This is one of the most common Git interview questions. The difference is subtle but important.

git fetch

Downloads changes from the remote but does NOT apply them to your working files. It only updates Git's knowledge of what's on the remote. Your local files are untouched.

Use it when you want to see what's new on GitHub before deciding whether to apply it.

git fetch origin
From github.com:DARKZONE17/crypto-tracker
   a3f5d9c..b1e2c3d  main → origin/main
# GitHub has new commits, but your files haven't changed
# To see what changed:
git log HEAD..origin/main --oneline
git pull

Downloads changes from the remote AND immediately applies them to your working branch. It is equivalent to running git fetch followed by git merge.

Use this in the normal daily workflow — it's what you run at the start of every session.

git pull origin main
From github.com:DARKZONE17/crypto-tracker
Updating a3f5d9c..b1e2c3d
Fast-forward
 kafka.html | 45 +++++
# = git fetch + git merge in one command
# your local files are now updated
SIMPLE WAY TO REMEMBER git fetch = download but don't apply (like downloading a file but not opening it).
git pull = download AND apply (like downloading and immediately installing it). Use pull in daily work.
// Chapter 10 — The Most Useful Survival Skills

Undo Mistakes — Getting Out of Trouble

This section will save you many times. Every developer makes mistakes — the skill is knowing how to undo them cleanly.

SCENARIO 1 — UNDO git add (UNSTAGE A FILE)

You ran git add file.py but changed your mind — you don't want that file in the next commit.

Terminal
git reset file.py # removes file.py from staging — your changes to the file are kept git reset # unstage EVERYTHING — none of it will be committed # your actual file changes are NOT deleted, just un-staged
SCENARIO 2 — UNDO A COMMIT (KEEP CHANGES)

You committed too early, or wrote the wrong message, or committed the wrong files. --soft undoes the commit but keeps your changes staged and ready to re-commit.

Terminal
git reset --soft HEAD~1 # HEAD~1 means "one commit before current" # --soft: undo the commit, keep changes STAGED # Your files are unchanged, staging is unchanged, commit is gone # Now you can fix the files / message and recommit
SCENARIO 3 — UNDO A COMMIT AND DELETE THE CHANGES

You committed something you want completely gone — bad code, secrets, whatever. --hard is the nuclear option: the commit AND the changes disappear. This cannot be undone.

Terminal
git reset --hard HEAD~1 # --hard: undo the commit AND discard all changes # Files go back to the state of the previous commit # ⚠ WARNING: changes are GONE. Cannot recover them. # Only use this on commits that have NOT been pushed to GitHub
SCENARIO 4 — RESTORE A SINGLE FILE

You edited a file and messed it up. You want it back to how it was in the last commit — discarding all your changes to that file.

Terminal
git checkout -- producer.py # classic: restores file to last committed state git restore producer.py # modern alternative (Git 2.23+) # ⚠ Your changes to that file are GONE immediately — no confirmation
SCENARIO 5 — UNDO A COMMIT THAT'S ALREADY ON GITHUB

You pushed a bad commit to GitHub. You can't just reset --hard because the commit exists on the remote. Instead, use git revert — it creates a new commit that undoes the old one, preserving history.

Terminal
git log --oneline b1e2c3d accidentally committed secrets ← this one a3f5d9c feat: add Kafka producer git revert b1e2c3d Revert "accidentally committed secrets" [main d4e5f6g] Revert "accidentally committed secrets" git push origin main # Creates a NEW commit that undoes b1e2c3d — safe to push
SituationCommandWhat Happens to Your Changes
git reset file.pyUnstage a fileChanges kept in working dir
git reset --soft HEAD~1Undo last commit, keep changes stagedChanges kept and staged
git reset HEAD~1Undo last commit, keep changes unstagedChanges kept but unstaged
git reset --hard HEAD~1Undo last commit, delete changesChanges GONE permanently
git checkout -- file.pyRestore one file to last commit stateChanges to that file GONE

// Chapter 11 — How Git Works Under the Hood

Git Internals — Snapshots, Not Differences

Most version control systems store the differences between file versions. Git does something different and more powerful — it stores complete snapshots.

SNAPSHOTS vs DIFFERENCES

Other VCS (SVN, CVS): Store "delta files" — only what changed. To reconstruct version 5, they apply change 1 + change 2 + change 3 + change 4 + change 5. Gets slow as history grows.

Git: Each commit stores a complete snapshot of every file. To see version 5, Git just loads the snapshot for commit 5. Instantly. This is why Git is so fast.

To save space, Git is smart: if a file didn't change between commit 4 and commit 5, commit 5's snapshot just stores a reference (pointer) to commit 4's version of that file instead of copying it again.

THE COMMIT HASH — GIT'S UNIQUE ID SYSTEM

Every commit has a unique 40-character identifier called a SHA-1 hash. It looks like: a3f5d9c7b1e2f4d8c9e0a1b2c3d4e5f6a7b8c9d0. You usually only need the first 7 characters to identify a commit uniquely.

The hash is not random — it's calculated from the commit's content, author, timestamp, and the hash of the previous commit. This means if anyone tampers with Git history, all hashes change. Git's integrity is mathematically guaranteed.

# Short hash (7 chars) is enough — used in git log --oneline
a3f5d9c feat: add Kafka producer

# Full hash — used for precise operations
a3f5d9c7b1e2f4d8c9e0a1b2c3d4e5f6a7b8c9d0

# HEAD is a pointer to your current position in history
# HEAD~1 = one commit before HEAD
# HEAD~3 = three commits before HEAD
// Chapter 12 — Advanced History Management

Rebase vs Merge — Clean History vs Full History

Both merge and rebase combine branches. They produce the same final code — but the commit history looks different. This is a frequent interview topic and a real daily decision in professional teams.

git merge — KEEPS FULL HISTORY

Creates a "merge commit" that joins two branches. The history shows exactly what happened — including the branch divergence and when it was merged. History is truthful but can look messy with many branches.

Before: main: A → B → C feature: ↓ D → E After git merge: main: A → B → C → M ← merge commit ↑ ↑ feature: D → E History is preserved exactly.
git checkout main
git merge feature/websocket
git rebase — LINEAR HISTORY

Replays your feature branch commits on top of main, as if you had branched off at the latest commit. The history looks like everything happened in a straight line — easier to read, but it rewrites commit hashes.

Before: main: A → B → C feature: ↓ D → E After git rebase main (on feature branch): main: A → B → C feature: ↓ D' → E' ← new hashes (rebased) History looks clean and linear.
git checkout feature/websocket
git rebase main
# then merge (will be fast-forward, no merge commit)
🧠 Analogy — Merge vs Rebase Merge = honest historian. Records every branch, every detour, every decision. The full story is there.
Rebase = editor who rewrites history to look clean. All commits appear in a straight line. Great for readability, but rewrites the past.
GOLDEN RULE OF REBASE Never rebase commits that have already been pushed to a shared remote. Rebase rewrites commit hashes. If teammates have pulled your old commits, rebasing creates conflicting histories for everyone. Rebase is safe on local-only branches. On shared branches — use merge.
INTERACTIVE REBASE — SQUASH COMMITS

Interactive rebase lets you clean up your commit history before merging — combine multiple small commits into one clean commit. Useful when you've made 10 "fix typo" commits and want to squash them into one meaningful commit.

# Squash the last 3 commits into one
git rebase -i HEAD~3
# Git opens your text editor with a list like:
pick a3f5d9c feat: add Kafka producer
squash b1e2c3d fix: typo in topic name
squash c4d5e6f fix: another typo
# Change 'pick' to 'squash' (or 's') on commits to combine
# Save → Git squashes them → asks you to write one combined message
// Chapter 13 — Temporary Saves

git stash — Save Work Without Committing

Stash is like a temporary clipboard for uncommitted changes. Use it when you need to switch branches but you're not ready to commit your current work.

WHEN AND HOW TO USE STASH

The scenario: You're halfway through adding WebSocket support on your feature branch. Suddenly you need to switch to main to fix an urgent bug. You can't commit your half-finished code (it doesn't even work yet). You can't just switch branches — Git will warn you about unsaved changes.

The solution: Stash your changes, switch branches, fix the bug, come back, pop the stash.

Terminal
# You're on feature/websocket with uncommitted changes git stash Saved working directory and index state WIP on feature/websocket # Your changes are gone from the working dir — but saved in the stash git stash save "half-done websocket reconnect logic" # Better: give the stash a description so you remember what it is # Now switch, do your work, come back git checkout main # ... fix the urgent bug, commit it ... git checkout feature/websocket # Restore your stashed changes git stash pop On branch feature/websocket Changes not staged for commit: modified: ws_producer.py # Your changes are back, stash is deleted # ── Other stash commands ─────────────────────────────────── git stash list # see all stashes stash@{0}: WIP on feature/websocket: half-done websocket reconnect logic git stash apply stash@{0} # apply a specific stash (but keep it in stash list) git stash drop stash@{0} # delete a specific stash git stash clear # delete ALL stashes
// Chapter 14 — Marking Releases

Tags — Marking Versions

Tags are labels you put on specific commits to mark important moments — usually release versions. Unlike branches, tags don't move when you add new commits. They permanently point to one commit.

CREATING AND USING TAGS
Terminal
# ── Lightweight tag — just a name pointing to a commit ──── git tag v1.0 # tags the current (latest) commit as v1.0 # ── Annotated tag — includes message, author, date ───────── git tag -a v1.0 -m "First working pipeline — Kafka producer + consumer" # use annotated tags for real releases (preferred) # ── Tag a past commit ────────────────────────────────────── git tag -a v0.9 a3f5d9c -m "Beta release" # tag a specific commit by its hash # ── List, view, push tags ────────────────────────────────── git tag # list all tags v0.9 v1.0 git show v1.0 # see what the tag points to git push origin v1.0 # push a tag to GitHub (tags aren't pushed by default) git push origin --tags # push ALL tags to GitHub
SEMANTIC VERSIONING Follow the convention: v1.0.0 (MAJOR.MINOR.PATCH).
MAJOR: breaking change. MINOR: new feature, backward compatible. PATCH: bug fix.
So v1.2.3 → v1.2.4 = bug fix, v1.2.3 → v1.3.0 = new feature, v1.2.3 → v2.0.0 = breaking change.

// Chapter 16 — Power User Commands

Advanced Commands

These commands are used less often but are extremely useful when you need them. Learn them once you're comfortable with the basics.

CHERRY-PICK — COPY A SPECIFIC COMMIT

Apply a specific commit from one branch onto another — without merging the entire branch. Useful when you want one specific fix but not everything else in that branch.

# Scenario: feature/websocket has a bug fix you need in main NOW
# but the whole feature isn't ready to merge yet
git log feature/websocket --oneline
d4e5f6g fix: handle Binance rate limit (you want JUST this one)
c3d4e5f feat: add reconnect logic
b2c3d4e feat: connect to Binance stream

git checkout main
git cherry-pick d4e5f6g
[main e5f6g7h] fix: handle Binance rate limit
# Only that one commit is now on main. The rest of the feature stays separate.
GIT HOOKS — AUTOMATE ACTIONS

Hooks are scripts that Git runs automatically before or after specific events. They're stored in .git/hooks/. Use them to enforce code quality rules automatically.

# Common hooks:
# pre-commit   → runs before a commit is created
# commit-msg   → validates commit message format
# pre-push     → runs before pushing to remote

# Example: .git/hooks/pre-commit (make it executable: chmod +x)
#!/bin/bash
python -m pytest tests/           # run tests before every commit
if [ $? -ne 0 ]; then
    echo "❌ Tests failed. Commit aborted."
    exit 1
fi
SUBMODULES — REPO INSIDE A REPO

Submodules let you include one Git repo inside another. Useful when your project depends on a specific version of another project that you don't own.

# Add a submodule
git submodule add https://github.com/other/library.git libs/library
Cloning into 'libs/library'...

# Clone a repo WITH its submodules
git clone --recurse-submodules https://github.com/org/project.git

# Update all submodules to latest
git submodule update --remote --merge
// Chapter 17 — Learn from Others' Mistakes

Common Beginner Mistakes — And How to Avoid Them

❌ COMMITTING EVERYTHING BLINDLY

Running git add . and git commit without checking what you're committing. You might accidentally commit secrets, binary files, debug code, or node_modules/.

Fix: Always run git status and git diff --staged before committing. Create a .gitignore before your first commit.

❌ WORKING DIRECTLY ON MAIN

Making all your changes directly on the main branch. If something breaks, your only working codebase is now broken.

Fix: Main should always be stable. Create a branch for every piece of work, no matter how small. It takes 3 seconds and saves hours of pain.

❌ NOT PULLING BEFORE PUSHING

You push your commits but GitHub rejects it because a teammate pushed first. Now you have a confusing error about "non-fast-forward updates".

Fix: Always git pull origin main before git push. Pull first, push second. If there are conflicts, resolve them locally before pushing.

❌ FEAR OF CONFLICTS

Avoiding branching because "merge conflicts are scary". So you work on main, rush commits to avoid others touching the same files, and create a messier situation.

Fix: Conflicts are normal. They take 2 minutes to resolve. Practice resolving them deliberately — create a conflict intentionally, resolve it, and you'll see they're not scary at all.

❌ COMMITTING SECRETS AND API KEYS

Accidentally committing .env files, API keys, database passwords to a public GitHub repo. These get found by bots within minutes and your keys get abused.

Fix: Create .gitignore before the first commit. Never hardcode secrets in code. If you accidentally commit a secret — rotate it immediately, it's already compromised.

❌ WRITING USELESS COMMIT MESSAGES

Messages like "fix", "update", "stuff", "asdf". Six months later you can't understand your own history — what changed, what broke, what was the context?

Fix: Write messages like: feat: add Kafka producer for Binance prices. Follow the format: type: short description of what and why.

// Chapter 18 — Your Learning Path

Practice Plan — 5 Days to Git Fluency

Git is learned by doing, not reading. Follow this plan and after 5 days you'll be comfortable with Git in real projects.

DAY 1 — The Basic Loop
  • Create a new folder, run git init
  • Create a few text files, stage them with git add
  • Make 5 commits with clear messages
  • Run git log --oneline and see your history
  • Change a file, run git diff, stage, commit
  • Goal: the add → commit loop feels automatic
DAY 2 — Branches
  • Create a branch with git checkout -b feature/test
  • Make 3 commits on that branch
  • Switch back to main — notice those commits are gone
  • Merge the feature branch into main
  • Delete the feature branch
  • Goal: branching feels normal and safe
DAY 3 — Conflicts
  • Create two branches from main
  • Edit the SAME line in BOTH branches
  • Merge one into main (succeeds)
  • Merge the other into main (conflict!)
  • Resolve the conflict manually, commit
  • Goal: conflicts are not scary anymore
DAY 4 — Remote / GitHub
  • Create a GitHub account and repo
  • Connect your local repo with git remote add origin
  • Push with git push origin main
  • Edit on GitHub, then git pull locally
  • Push a feature branch, open a PR, merge it
  • Goal: comfortable with push/pull/PR
DAY 5 — Undo and Rebase
  • Make a commit, then undo it with reset --soft
  • Make a commit, then undo it with reset --hard
  • Use git stash to save half-done work, switch, come back
  • Make 3 small commits, squash them with rebase -i
  • Tag a commit with git tag v1.0
  • Goal: confident undoing mistakes
AFTER DAY 5 — USE GIT DAILY

Use Git for every project, even personal ones. The commands become automatic through repetition. After 2–3 weeks of daily use, you'll stop thinking about Git commands and just use them naturally.

Apply Git to this crypto-tracker project — use the branch naming conventions in this guide, write good commit messages, push to GitHub daily.
