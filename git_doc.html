Foundation

What is Git?

Git is a version control system. It tracks every change ever made to your code, stores a complete history, and lets you go back to any previous state. Think of it as unlimited undo with a detailed history of who changed what, when, and why.

The Problem Git Solves

Without Git, most developers end up with folders like: project_v1, project_v2, project_final, project_final_REAL, project_DONT_DELETE. This is manual version control — it's messy, error-prone, and makes collaboration impossible.

With Git, every saved state (called a commit) is tracked automatically with a timestamp, a message describing the change, and the exact lines that changed. You can jump back to any commit instantly. You can see the complete history of your project in one command.

Clarification

Git vs GitHub — They Are Not the Same

Many beginners confuse Git and GitHub. They are completely different things that happen to work together.

Git

Software that runs on YOUR laptop. Tracks file changes locally. Works with zero internet connection. Free and open source. Created by Linus Torvalds (creator of Linux) in 2005. A command-line tool you install once.

When you run git commit, that's Git working locally on your machine — nothing goes to the internet.

GitHub

A website that hosts Git repositories in the cloud. Provides a web UI to browse code, review changes, and manage collaboration. Owned by Microsoft. Requires an account. Not the only option — there's also GitLab, Bitbucket, etc.

When you run git push, that's sending your local Git history up to GitHub in the cloud.

💡
Key relationship Git is the tool. GitHub is cloud storage for that tool. You can use Git without GitHub (local only, no backup). You cannot use GitHub without Git. For this project: Git tracks your code locally, GitHub stores a backup and lets you navigate your history on the web.
The Most Important Concept

Why Branches? What Goes Wrong Without Them

Branches are what make Git powerful for real development. Without branches, adding any new feature risks breaking what already works.

Without Branches — What Goes Wrong

Imagine you're adding WebSocket functionality to your crypto tracker. While coding, you break the Kafka producer you already had working. Without branches, your main branch is now broken — and there's no easy way to isolate your broken WebSocket code from your working Kafka code.

You can't commit your broken work (breaks the repo), can't deploy (broken), and can't easily go back without losing your WebSocket changes. This is exactly the situation branches solve.

With Branches — Complete Isolation

You create a branch called feature/websocket and do all your WebSocket work there. Your main branch still has your working Kafka producer, untouched. You can break things in your feature branch all you want — it doesn't affect main. When the WebSocket code works, you merge it into main.

main: ●───────────────────────────────────────────────● ← stable at all times │ ↑ │ branch off here merge here │ ↓ │ feature: ●──●──●──●──●──● ← your work lives here, safely isolated │ │ │ │ │ └── fix: handle connection timeout │ │ │ │ └───── feat: add message parsing │ │ │ └──────── feat: connect to Binance stream │ │ └─────────── feat: install websockets lib │ └────────────── chore: create ws_producer.py └───────────────── branch created from main
✅
The golden rule of branches main should always be in a working, deployable state. Never commit broken or experimental code to main. Always work in a feature branch. Merge to main only when the feature is done and tested.

Branch Naming Convention

Use the format type/short-description. The type tells you what kind of change it is:

feature/kafka-producer ← adding new functionality feature/websocket-binance ← adding new functionality fix/consumer-offset-bug ← fixing a bug fix/docker-port-conflict ← fixing a bug docs/architecture-update ← documentation changes refactor/config-parametrize ← restructuring code without changing behavior chore/update-dependencies ← maintenance tasks
One-Time Setup

Git Setup

Step 1 — Install Git

git --version git version 2.39.0 ← if you see this, Git is already installed
If not installed:
winget install Git.Git # Windows brew install git # Mac sudo apt install git # Ubuntu/Linux

Step 2 — Set Your Identity (required once)

Git attaches your name and email to every commit you make. This is how you know who made each change.

git config --global user.name "Your Name" git config --global user.email "your@email.com" git config --global init.defaultBranch main # The above makes new repos start on 'main' (modern standard)
Verify your config
git config --list user.name=Your Name user.email=your@email.com
⚠️
Without setting user.name and user.email, commits won't have an author, and GitHub won't credit them to your account profile.

Every Work Session

Daily Git Workflow — The Loop You Run Every Time

This is the exact sequence of commands you run every time you sit down to work. Same order, every time. Internalize this loop and Git becomes muscle memory.

The Complete Daily Workflow

STEP 1 — Always start by getting the latest code from GitHub
git pull origin main Already up to date. ← no changes since last time OR Fast-forward ← pulled new commits (someone else pushed, or you pushed from another machine)
STEP 2 — Create or switch to your feature branch
git checkout -b feature/websocket-binance Switched to a new branch 'feature/websocket-binance' # OR, if the branch already exists: git checkout feature/websocket-binance
STEP 3 — Write your code... then check what changed
git status On branch feature/websocket-binance Changes not staged for commit: modified: Websocket.html Untracked files: ws_producer.py
STEP 4 — Stage the changes you want to commit
git add Websocket.html ws_producer.py # Adds those two files to the staging area git add . # OR add everything at once
STEP 5 — Commit with a descriptive message
git commit -m "feat: add Binance WebSocket producer" [feature/websocket-binance 3a4b5c6] feat: add Binance WebSocket producer 2 files changed, 87 insertions(+)
STEP 6 — Push your branch to GitHub
git push origin feature/websocket-binance Enumerating objects: 5, done. Branch 'feature/websocket-binance' set up to track remote branch.

Good vs Bad Commit Messages

Commit messages are the story of your project. Future-you will thank present-you for writing clear messages.

❌ Bad — useless

git commit -m "stuff"
git commit -m "fix"
git commit -m "asdf"
git commit -m "changes"
git commit -m "working now"

✅ Good — tells a story

feat: add Binance WebSocket producer
fix: resolve NoBrokersAvailable on startup
docs: update Kafka offset explanation
refactor: extract config to config.py
chore: add .env to .gitignore

Format: type: short description  —  Types: feat fix docs refactor chore test

Combining Branches

Merging a Branch into Main

Merge via Command Line

git checkout main # switch to main first git pull origin main # get latest main from GitHub git merge feature/websocket-binance # merge your feature branch in Updating e5f6g7h..a1b2c3d Fast-forward ws_producer.py | 87 ++++++++++++++++++ git push origin main # push the merged result to GitHub
💡
In a real workflow, you'd usually merge via a Pull Request on GitHub rather than locally — it creates a record of the merge and allows code review. For a solo project, local merge is perfectly fine.
When Two Changes Clash

Merge Conflicts — What They Are and How to Fix Them

What is a Merge Conflict?

A conflict happens when two branches both changed the same line of the same file differently. Git can't decide which version is correct — so it marks the conflict in the file and asks you to decide.

git merge feature/websocket-binance CONFLICT (content): Merge conflict in producer.py Automatic merge failed; fix conflicts and then commit the result.

What the Conflicted File Looks Like

Git adds conflict markers to the file. Everything between <<<<<<< HEAD and ======= is your version. Everything between ======= and >>>>>>> is the incoming branch's version.

# producer.py — after merge conflict

<<<<<<< HEAD
TOPIC = "crypto-prices"         # YOUR version (current branch)
=======
TOPIC = "crypto-prices-dev"     # INCOMING version (feature branch)
>>>>>>> feature/websocket-binance

Open the file, decide which version to keep (or write a combination), delete the conflict markers, then:

git add producer.py # mark conflict as resolved git commit -m "merge: resolve topic name conflict"
✅
VS Code has excellent built-in conflict resolution. When you open a conflicted file, it shows "Accept Current Change" / "Accept Incoming Change" / "Accept Both Changes" buttons above each conflict. Click the right one and the markers are handled automatically.
Remote Repository

GitHub — Pull Requests

Pull Request Workflow

A Pull Request (PR) is the formal way to merge code on GitHub. It shows a diff of all your changes, allows code review, and creates a record of what was merged and why.

1. Push your feature branch to GitHub
git push origin feature/websocket-binance

2. Go to github.com/DARKZONE17/crypto-tracker — you'll see a yellow banner: "Compare & pull request" — click it.

3. Fill in: Title (what did you build), Description (why, how to test). Set Base: main, Compare: feature/websocket-binance.

4. Review the diff — check your own changes look correct. Then click "Merge pull request".

5. Back in terminal — sync your local main with the merged result
git checkout main git pull origin main
What Not to Track

.gitignore — Never Commit These

.gitignore for This Project

The .gitignore file tells Git which files and folders to completely ignore. These are files you never want tracked or backed up to GitHub.

# .gitignore — place in your project root

# Environment variables and secrets — NEVER commit these
.env
.env.local
.env.*.local
secrets.py
config_secrets.py

# Python artifacts — auto-generated, don't need to track
__pycache__/
*.pyc
*.pyo
.venv/
venv/
dist/
*.egg-info/

# macOS system files
.DS_Store

# Windows system files
Thumbs.db

# IDE config (optional — some teams track these)
.vscode/settings.json
.idea/
🚨
If you accidentally commit a secret key or password Remove the file, commit the removal, AND rotate the exposed secret immediately. The secret is still visible in the Git history even after deletion — anyone who clones the repo can see old commits. You must invalidate the secret from the service that issued it (Binance API key, AWS key, etc.).
