Component 01

Binance WebSocket

Your data source. This is where real-world crypto trades enter your system.

🔌
What WebSocket actually is
wss://stream.binance.com:9443/ws/btcusdt@trade
Real-Time Source ▶
What it does
Binance keeps a persistent open connection to your Python script. Like a phone call that never hangs up.
Every time any trade happens on BTCUSDT, Binance instantly pushes a JSON message to you. You don't ask — it just arrives.
No API key needed for public trade streams. Free, unlimited events, no rate limiting.
Latency is <100ms. By the time your script sees the message, the trade just happened on Binance's servers.
Why NOT REST API (your old way)
With REST you poll every N seconds — "hey, what happened?" You were limited to 10–20 events/min.
Between your polls, hundreds of trades happen and you miss them. You had to batch them — not real-time.
Every poll = new HTTP connection overhead (TCP handshake, headers, response). Wasteful.
WebSocket = one connection, infinite events. Server pushes to you. Zero polling.
What a Binance trade message looks like (raw JSON)
{
  "e": "trade",              ← event type
  "E": 1718000000000,      ← event time (when Binance sent it)
  "s": "BTCUSDT",          ← trading pair / symbol
  "t": 3456789,            ← trade ID — unique per trade ⭐
  "p": "67432.50",         ← price in USDT
  "q": "0.00142",          ← quantity of BTC traded
  "T": 1718000000001,      ← trade time ⭐ USE THIS as event_time
  "m": false               ← was buyer the market maker?
}
💡 Critical: Always use field "T" (trade time) as your event_time — NOT time.time() on your machine. If your server is slow or restarts, time.time() would be wrong. "T" is when the trade actually happened on Binance.
Reconnect + Backoff Logic (what your code must do)
1
Connect to Binance WS
Open wss://stream.binance.com:9443/ws/btcusdt@trade. If it connects → start receiving messages. Move to step 2.
2
Receive messages in async loop
async for message in websocket: — each message is one trade. Parse it, extract fields, push to Kafka.
3
Connection drops (it will happen)
Network hiccup, Binance restarts, your server sleeps. The loop throws a ConnectionClosedError.
4
Exponential backoff — wait, then retry
Attempt 1: wait 1s. Attempt 2: wait 2s. Attempt 3: wait 4s. Attempt 4: wait 8s. Attempt 5: wait 16s. Why? If Binance is having issues, hammering it immediately makes things worse for everyone.
5
After N failures → DLQ + alert
If you've retried 10 times and still can't connect, write an error event to a Dead Letter Queue topic in Kafka and fire an alert. Don't loop forever silently.
Your ws_producer.py — skeleton
import asyncio, json, websockets
from confluent_kafka import Producer

producer = Producer({
    'bootstrap.servers': 'localhost:9092',
    'enable.idempotence': 'true',   # prevents duplicate writes on retry
    'acks': 'all',
    'compression.type': 'snappy',
})

async def connect_and_stream():
    backoff = 1
    attempts = 0
    while attempts < 10:
        try:
            async with websockets.connect(
                "wss://stream.binance.com:9443/ws/btcusdt@trade"
            ) as ws:
                backoff = 1          # reset on successful connect
                attempts = 0
                async for raw in ws:
                    msg = json.loads(raw)
                    payload = {
                        "coin":       msg["s"],
                        "price":      float(msg["p"]),
                        "quantity":   float(msg["q"]),
                        "trade_id":   msg["t"],
                        "event_time": msg["T"],  # ⭐ NOT time.time()
                    }
                    producer.produce(
                        topic="crypto-trades",
                        key=payload["coin"],         # partition key
                        value=json.dumps(payload).encode(),
                    )
                    producer.poll(0)                 # ⭐ don't forget this
        except Exception as e:
            attempts += 1
            print(f"Attempt {attempts} failed: {e}. Retry in {backoff}s")
            await asyncio.sleep(backoff)
            backoff = min(backoff * 2, 16)     # cap at 16s
    producer.flush()                                  # ⭐ flush on exit
Component 02

Apache Kafka

Your shock absorber. Kafka sits between your WS producer and Spark, so if Spark is slow or restarts, no messages are lost.

📨
Kafka — the distributed commit log
topic: crypto-trades · 6 partitions · replication: 2
Message Bus ▶
The real-world analogy
Kafka is like a conveyor belt in a factory. Your WS producer puts items on the belt. Spark picks them up. The belt keeps moving even if Spark pauses temporarily.
Messages are stored on disk for 7 days. If Spark crashes at 3am, when it restarts it replays from where it left off — no data lost.
Multiple consumers can read the same topic independently. Later you can add an alerts service alongside Spark — no code change to producer.
Core concepts you must know
Topic = named channel. Your topic is crypto-trades. Think of it like a database table for messages.
Partition = a topic is split into 6 buckets. All BTC messages go to one bucket, ETH to another (because key = coin name). This enables parallel processing.
Offset = position number of a message within a partition. Spark tracks its offset so it knows where to resume after a restart.
Consumer group = Spark reads as a group. Each partition is assigned to one Spark executor — parallel processing without duplicate reads.
How partitioning works for your 3 coins
BTCUSDT → Partition 0
offset 0: {price: 67432}
offset 1: {price: 67435}
offset 2: {price: 67431}
offset 3: ← next
ETHUSDT → Partition 1
offset 0: {price: 3521}
offset 1: {price: 3519}
offset 2: ← next
SOLUSDT → Partition 2
offset 0: {price: 142}
offset 1: {price: 143}
offset 2: {price: 141}
offset 3: ← next
All BTC messages always go to the same partition → ordering guaranteed per coin
⚠️ The #1 gotcha with confluent-kafka: You MUST call producer.poll(0) in your message loop AND producer.flush() on shutdown. Without poll(0), messages sit in an internal buffer and Kafka never sees them. This silently loses data. Every senior engineer forgets this once.
Your Kafka topic config (docker-compose or admin client)
topic:              crypto-trades
num.partitions:     6          # 6 = room to grow to 6 coins
replication.factor: 2          # 2 copies of every message
retention.ms:       604800000  # 7 days = 604800 seconds

Producer settings:
  acks:               all        # wait for ALL replicas to confirm write
  enable.idempotence: true       # no duplicate writes if producer retries
  compression.type:   snappy     # ~50% smaller messages, faster transfer
  linger.ms:          5          # wait 5ms to batch messages together
  batch.size:         16384      # max 16KB per batch
Component 03

Databricks Structured Streaming

Your processing engine. Reads from Kafka, cleans data, computes rolling windows, detects anomalies — continuously, every 5 seconds.

⚡
Spark Structured Streaming
micro-batch · 5s trigger · watermark: 10min · checkpoint: /mnt/checkpoints/crypto
Processing Engine ▶
What Spark does here
Reads messages from Kafka every 5 seconds (micro-batch). Treats the stream like an infinite table that keeps growing.
Parses the JSON, applies your schema, computes 10-minute rolling windows — avg price, stddev, count of trades.
Flags anomalies with Z-score logic, then writes results to Delta Lake.
If it crashes, it resumes from checkpoint — knows exactly which Kafka offset it last processed.
3 critical concepts for interviews
Watermark (10 min) = "I'll wait up to 10 minutes for late data, then discard it." Without this, Spark holds state forever and runs out of memory.
Checkpoint = Spark saves its progress (offsets + window state) to disk every batch. Delete it and Spark replays everything from the start.
maxOffsetsPerTrigger = limits how many Kafka messages Spark reads per batch. Prevents Spark from being overwhelmed if Kafka has a backlog.
Late data — why watermark is needed
NORMAL: arrives on time
Trade at 10:00:00 → arrives at Spark at 10:00:01 ✅
Goes into 10:00–10:10 window. Fine.
LATE: network delay
Trade at 10:00:00 → arrives at Spark at 10:07:00 ⚠️
7 min late — still within 10min watermark. Included.
TOO LATE: dropped
Trade at 10:00:00 → arrives at Spark at 10:15:00 ❌
15 min late — exceeds watermark. Discarded.
Your spark_stream.py — key parts
from pyspark.sql import SparkSession
from pyspark.sql.functions import from_json, col, window, avg, stddev
from pyspark.sql.types import StructType, StringType, DoubleType, LongType

spark = SparkSession.builder.appName("CryptoStream").getOrCreate()

# ① Always define schema explicitly — NEVER infer
schema = StructType() \
  .add("coin",       StringType()) \
  .add("price",      DoubleType()) \
  .add("quantity",   DoubleType()) \
  .add("trade_id",   LongType()) \
  .add("event_time", LongType())

# ② Read from Kafka
raw = spark.readStream.format("kafka") \
  .option("kafka.bootstrap.servers", "localhost:9092") \
  .option("subscribe", "crypto-trades") \
  .option("startingOffsets", "latest") \
  .option("maxOffsetsPerTrigger", 10000) \
  .load()

# ③ Parse JSON + cast event_time to timestamp
parsed = raw.select(from_json(col("value").cast("string"), schema).alias("d")) \
  .select("d.*") \
  .withColumn("event_time", (col("event_time") / 1000).cast("timestamp"))

# ④ Watermark + dedup
clean = parsed \
  .withWatermark("event_time", "10 minutes") \
  .dropDuplicates(["trade_id"])

# ⑤ 10-min sliding window (slide every 5 min)
windowed = clean.groupBy(
  window(col("event_time"), "10 minutes", "5 minutes"),
  col("coin")
).agg(
  avg("price").alias("avg_price"),
  stddev("price").alias("stddev_price"),
)

# ⑥ Write to Delta with checkpoint
windowed.writeStream \
  .format("delta") \
  .outputMode("append") \
  .option("checkpointLocation", "/mnt/checkpoints/crypto") \
  .trigger(processingTime="5 seconds") \
  .start("/mnt/delta/silver")
Component 04

Delta Lake — Bronze → Silver → Gold

Your storage layer. Three tiers of data quality — raw, cleaned, and business-ready. Each layer has a clear purpose.

🗄
Medallion Architecture
Bronze (raw) → Silver (clean) → Gold (aggregated + anomalies)
Storage Layer ▶
Why Delta over plain Parquet files?
FeaturePlain ParquetDelta Lake
ACID transactions❌ No — partial writes corrupt data✅ Yes — atomic commits
Time travel❌ Overwrite = gone forever✅ VERSION AS OF 1 — go back
Schema enforcement❌ Wrong column? File is corrupt✅ Rejects bad schema automatically
Upserts (MERGE)❌ Read all, rewrite all✅ MERGE INTO — update specific rows
Concurrent reads+writes❌ Unsafe✅ Optimistic concurrency control
🥉
Bronze
raw layer
Raw — exact copy of Kafka messages
Never transform here. Store exactly what came from Kafka, including Kafka metadata (offset, partition, topic). This is your audit log — if something goes wrong downstream, you can always replay from Bronze.
coin STRING, price STRING, quantity STRING, trade_id BIGINT,
event_time BIGINT, kafka_offset BIGINT, kafka_partition INT
🥈
Silver
clean layer
Cleaned — typed, deduped, validated
Parse strings to proper types, cast event_time to timestamp, drop duplicates on trade_id, filter out nulls and invalid prices. This is the "trusted" dataset that analytics queries should use.
coin STRING, price DOUBLE, quantity DOUBLE, trade_id BIGINT,
event_time TIMESTAMP, date DATE
🥇
Gold
business layer
Aggregated — windowed metrics + anomaly flag
10-minute rolling average and stddev per coin, z-score calculation, is_anomaly boolean flag. This is what Grafana queries. Partitioned by coin + date for fast lookups.
window_start TIMESTAMP, window_end TIMESTAMP, coin STRING,
avg_price DOUBLE, stddev_price DOUBLE, z_score DOUBLE,
is_anomaly BOOLEAN, trade_count BIGINT
🔵 MERGE explained simply: When Spark writes a new 10-min window result, it doesn't append a duplicate — it says "if a row with this window + coin already exists, UPDATE it; otherwise INSERT it." This is how you get clean aggregation results without duplicates.
Time travel — your secret interview weapon
-- What was the avg BTC price at version 1 of this table?
SELECT * FROM crypto_gold VERSION AS OF 1

-- What was the data 2 hours ago?
SELECT * FROM crypto_gold TIMESTAMP AS OF '2024-06-10 10:00:00'

-- Optimize + Z-order for fast queries by coin + timestamp
OPTIMIZE crypto_gold ZORDER BY (coin, window_start)

-- Clean old file versions (keep 7 days)

The Logic Inside Gold Layer

Z-Score Anomaly Detection

Why you chose Z-score over ML, and how to explain it in 30 seconds.

🎯
Z-Score = (price − mean) / stddev
threshold: abs(z) > 2.0 → is_anomaly = true
Anomaly Detection ▶
What Z-score means
Z-score measures how many standard deviations away a price is from the rolling average. Z=0 means exactly average. Z=3 means 3 stddevs away from average — very unusual.
In a normal distribution, 95.4% of values fall within ±2 stddevs. If Z > 2.0, only 4.6% of normal trades would look like this. Flag it.
Example: rolling avg = $67,000, stddev = $200. A price of $67,500 has Z = (67500 − 67000) / 200 = 2.5 → anomaly.
Why NOT LSTM / ML models
You're a Data Engineer, not an ML engineer. Your job is the pipeline, not the model. Z-score is the right tool for the right role.
LSTM needs weeks of training data, GPU resources, retraining pipelines, model versioning. Overkill for this use case.
Z-score runs inside a Spark UDF — zero extra infrastructure, explainable in 30 seconds, fully deterministic. Perfect for a DE portfolio project.
Z-score as a Spark UDF
from pyspark.sql.functions import udf
from pyspark.sql.types import DoubleType

@udf(returnType=DoubleType())
def z_score(price, mean, stddev):
    if stddev is None or stddev == 0:
        return 0.0
    return abs((price - mean) / stddev)

gold = windowed.withColumn(
    "z_score", z_score(col("avg_price"), col("rolling_mean"), col("stddev_price"))
).withColumn(
    "is_anomaly", col("z_score") > 2.0
)
Putting It All Together

End-to-End Data Journey

One BTC trade happens. Here is exactly what your system does — step by step, millisecond by millisecond.

1
T+0ms — Trade happens on Binance
Someone buys 0.001 BTC at $67,432.50. Binance records this trade. Your WebSocket connection receives the JSON message within ~50ms.
{"s":"BTCUSDT","p":"67432.50","q":"0.001","t":3456789,"T":1718000000001}
2
T+50ms — ws_producer.py receives + parses
Your Python script wakes up (it's waiting in async for message in ws). It parses the JSON, extracts the fields, stamps with field "T" as event_time.
3
T+55ms — Pushed to Kafka partition 0
confluent-kafka producer sends the message to Kafka with key="BTCUSDT". Kafka routes it to partition 0 (all BTC goes here). acks=all means Kafka waits for both replicas to confirm. Kafka sends back a delivery confirmation.
4
T+55ms to T+5s — Kafka holds the message
The message sits in Kafka partition 0 at offset N. It will stay there for 7 days. Spark reads every 5 seconds (trigger interval), so it'll pick this up in the next batch — up to 5 seconds from now.
5
T+5s — Spark micro-batch fires
Spark wakes up, reads all new messages from Kafka since its last checkpoint offset. Parses JSON using your StructType schema. Applies watermark. Drops duplicates on trade_id. Computes 10-minute rolling window — avg price and stddev for BTCUSDT.
6
T+5s — Anomaly check runs
Z-score UDF fires: z = (67432.50 − rolling_avg) / rolling_stddev. If this price is unusual for the last 10 minutes, is_anomaly = true.
7
T+5s — Written to Delta Lake Gold
Spark writes the windowed result to Delta Lake Gold via MERGE. If the 10-min window for BTCUSDT already exists, it updates the row. If not, it inserts. Delta ACID ensures no partial writes. Checkpoint updated — this Kafka offset is now "done."
8
T+10s — Grafana refreshes dashboard
Grafana auto-queries Delta Lake Gold every 5 seconds. Panel 1 updates with new price point. If is_anomaly = true, Panel 3 alert table shows a new red row. Total end-to-end latency: ~10 seconds from trade to dashboard update.
Interview Prep

The 5 Questions You'll Be Asked

Know these cold. Say exactly this.

Q: Why WebSocket instead of REST?
"My original design used CoinGecko REST polling, which hit a rate limit of 10–20 events per minute and required batching — not real-time. WebSocket gives me a persistent TCP connection where Binance pushes every trade event as it happens, under 100ms latency, with no rate limit. The tradeoff is I need to manage reconnection logic, but that's handled with exponential backoff."
Q: What happens if Spark crashes at 3am?
"Nothing is lost. Kafka retains messages for 7 days with replication factor 2. Spark saves its last-processed Kafka offset to the checkpoint directory after every batch. When it restarts, it reads the checkpoint, sees offset N was the last processed, and resumes from offset N+1. The gap in processing time means slightly delayed data, but zero data loss."
Q: What is watermarking and why do you need it?
"Watermarking tells Spark how long to wait for late-arriving data before closing a time window. I set 10 minutes — meaning if a trade with event_time 10:00 arrives at 10:09, it's still included in the 10:00–10:10 window. Without watermarking, Spark keeps every window open forever waiting for late data, which grows state unboundedly and eventually causes out-of-memory errors. It also bounds the deduplication state."
Q: Why Bronze-Silver-Gold and not just one table?
"Separation of concerns. Bronze is my audit log — raw, untouched, append-only. If I introduce a bug in Silver transformation, I can replay from Bronze without re-fetching from Binance. Silver is the clean, trusted dataset that any analyst can query. Gold is purpose-built for my specific dashboard queries — pre-aggregated, anomaly-flagged, partitioned for performance. Each layer serves a different consumer."
Q: How does your pipeline scale to 100 coins?
"The WebSocket producer subscribes to a combined stream — Binance supports multi-stream URLs like /stream?streams=btcusdt@trade/ethusdt@trade/... or you can open parallel connections. Kafka already has 6–12 partitions so more coins just distribute across them — coin-as-key ensures ordering is preserved per coin. Spark partitions work is already parallelized across executors. The only change needed is adding more partitions to the Kafka topic and potentially increasing executor count in Databricks."
