Foundation

What is a WebSocket?

Before you read any code, understand the protocol. WebSocket is a two-way communication channel between your script and a server — established once, kept open forever.

HTTP request/response vs WebSocket persistent connection
The key insight: With HTTP you keep knocking on the door asking "any new trades?" With WebSocket, Binance calls you the instant a trade happens. You don't poll — you listen.

Implementation · Option 1

Callback-Based — Event-Driven Style

You define functions for each event. The library calls them automatically when that event occurs. You are reacting, not controlling.

The doorbell mental model

Real doorbell system

Someone presses bell 🔔
→ Ring triggers automatically
→ You didn't check manually
→ The event called your handler

Kafka callback style

Trade arrives 📦
→ on_message() fires automatically
→ You didn't poll Binance
→ The event called your handler

Install dependency

pip install websocket-client

Full code — Option 1

import websocket
import json

# ── Handler 1: Called when a message arrives ──────────────────────────
def on_message(ws, message):
    data = json.loads(message)   # raw JSON string → Python dict
    print("Received:", data)

# ── Handler 2: Called when an error occurs ────────────────────────────
def on_error(ws, error):
    print("Error:", error)

# ── Handler 3: Called when connection closes ──────────────────────────
def on_close(ws, close_status_code, close_msg):
    print("Closed connection")

# ── Handler 4: Called once connection opens ───────────────────────────
def on_open(ws):
    print("Connection opened")

# Binance trade stream URL
socket = "wss://stream.binance.com:9443/ws/btcusdt@trade"

# Create the WebSocket client, bind the handlers
ws = websocket.WebSocketApp(
    socket,
    on_message=on_message,   # message arrives → call this
    on_error=on_error,       # error occurs   → call this
    on_close=on_close        # closes         → call this
)

ws.on_open = on_open         # opens          → call this

# Start the WebSocket — this blocks forever, calling handlers as events fire
ws.run_forever()
📨
on_message(ws, message)
Triggered for every incoming trade. Receives raw JSON string. You parse and process it here.
⚠️
on_error(ws, error)
Network failures, invalid payloads, connection drops. Log here, handle retries outside.
🔌
on_close(ws, code, msg)
Server disconnect, network failure, manual termination. Cleanup and reconnect logic goes here.
🟢
on_open(ws)
Fires once when connection is ready. Good place to send a subscription message if needed.

Execution flow — Option 1

ws.run_forever() — starts the loop
↓ connection handshake with Binance
on_open() fires once
↓ Binance pushes a trade
on_message() fires → parse JSON → process
↓ repeats for every trade (could be 100s/sec)
on_message() fires again → parse → process
↓ if connection drops
on_error() fires → then on_close() fires
Implementation · Option 2

Async Loop — You Control the Flow

Instead of callbacks, you write an explicit async for loop. You decide what to do with each message, step by step. More readable, more flexible.

Install dependency

pip install websockets
Note: This is a different library than Option 1. websocket-client (Option 1) vs websockets (Option 2 — plural, different package).

Full code — Option 2

import asyncio
import json
import websockets

async def connect_and_stream():
    url = "wss://stream.binance.com:9443/ws/btcusdt@trade"

    print("Connecting...")

    # Opens connection → keeps it open → auto-closes if something crashes
    async with websockets.connect(url) as ws:
        print("Connection opened")

        # Infinite loop — yields each message as Binance pushes it
        async for message in ws:
            try:
                data = json.loads(message)   # JSON string → dict
                print("Received:", data)

            except json.JSONDecodeError as e:
                print("JSON error:", e)

# Start the async event loop and run the function
asyncio.run(connect_and_stream())

Step-by-step execution trace

↓ starts async event loop, enters function

↓ TCP + TLS handshake with Binance

↓ connection open, auto-cleanup registered
print("Connection opened")
↓ enters the infinite message loop

↓ loops back, waits for next trade

The async with context manager is important — if anything crashes, it automatically closes the WebSocket cleanly and releases network resources.
Core Concept

What Does "async" Actually Mean?

The hardest part for beginners. Once you truly get this, async makes perfect sense everywhere.

Blocking vs Non-Blocking — timeline view

📞 Blocking — Old way

You call a restaurant 📞
You stay on hold doing NOTHING
You can't make another call
You just wait... and wait...

CPU is stuck, wasting time

🍕 Async — Smart way

You order food online 🍕
You go do other work
You can handle other tasks
Restaurant notifies you when ready

CPU is free to do other things

The one line that matters

async for message in ws:
    # This line means:
    # "Stay idle until Binance sends a trade, then wake up instantly.
    #  While idle, DON'T block — let other coroutines run if needed."
async for is not a normal for loop. It doesn't spin checking "is there data yet?" It registers itself with the event loop and says "wake me when there's data." The event loop can handle other tasks meanwhile. When Binance pushes data → the loop resumes instantly.

async keywords demystified

KeywordWhat it meansWhen to use it
async def This function can be paused and resumed without blocking Any function that uses await inside it
await Pause here, let other tasks run, resume when this is done Calling another async function or coroutine
async for Iterate over an async stream — each item may require waiting Reading from WebSocket, async file I/O
async with Context manager that can pause on entry/exit Opening WebSocket connections, async DB sessions
asyncio.run() Create event loop and run one async function until complete Entry point for any async program

Each message from Binance represents one individual trade — a buyer and seller matched at a price. Here's exactly what arrives.

E Event time Unix timestamp (ms) when Binance created this event 1718000000001
s Symbol Trading pair — always uppercase "BTCUSDT"
t Trade ID Unique ID per trade — strictly increasing 3854291201

Accessing the useful fields in Python

def on_message(ws, message):
    data = json.loads(message)

    price    = float(data["p"])    # "67432.50" → 67432.5  ← most important
    quantity = float(data["q"])    # "0.00150"  → 0.0015
    symbol   = data["s"]            # "BTCUSDT"
    trade_id = data["t"]            # 3854291201
    trade_ts = data["T"]            # unix ms — convert: ts/1000
    is_sell  = data["m"]            # True = sell order, False = buy order

    usd_volume = price * quantity    # notional value traded in USD

    print(f"{symbol}: ${price:,.2f} × {quantity} BTC = ${usd_volume:,.2f}")
⚠ Price and quantity come as strings, not numbers. Always float(data["p"]) before doing math. This is a Binance design choice for decimal precision.

Pipeline Integration

WebSocket → Kafka Producer

The WebSocket gives you live data. The Kafka producer forwards it into your pipeline. Here's how they connect — and why confluent-kafka makes this clean.

Why confluent-kafka fits here

confluent-kafka has built-in internal buffering. When you call producer.produce(), messages go into an internal buffer (backed by librdkafka). The buffer batches them and sends to Kafka with configurable latency (linger.ms). This means you do not need to build your own buffer — just produce() and it handles the rest. Call producer.flush() when you want to force-send everything now.
For kafka-python users: This behavior is why confluent-kafka is preferred for high-frequency streams like this. kafka-python buffers too, but confluent-kafka's C-backed librdkafka is significantly faster and more production-proven at high message rates.
Full Pipeline — Binance WebSocket → confluent-kafka → Kafka Broker
1Option 1 (callbacks) → Kafka producer
import websocket
import json
from confluent_kafka import Producer

# Setup confluent-kafka producer
producer = Producer({
    'bootstrap.servers': 'localhost:9092'
})

def delivery_report(err, msg):
    if err:
        print(f"❌ Delivery failed: {err}")
    else:
        print(f"✓ Sent to partition {msg.partition()} @ offset {msg.offset()}")

def on_message(ws, message):
    data = json.loads(message)

    # Build the Kafka message payload
    kafka_msg = {
        "symbol":    data["s"],
        "price":     float(data["p"]),
        "quantity":  float(data["q"]),
        "trade_id":  data["t"],
        "timestamp": data["T"]
    }

    # produce() is non-blocking — goes to confluent-kafka's internal buffer
    # No need for a manual buffer — confluent-kafka handles batching for you
    producer.produce(
        topic='crypto-prices',
        key='BTC',
        value=json.dumps(kafka_msg).encode('utf-8'),
        on_delivery=delivery_report
    )
    # poll() processes callbacks — call regularly to avoid buffer buildup
    producer.poll(0)

def on_error(ws, error):   print("Error:", error)
def on_close(ws, c, m):
    producer.flush()   # flush remaining messages before closing
    print("Closed")
def on_open(ws):         print("Connected")

ws = websocket.WebSocketApp(
    "wss://stream.binance.com:9443/ws/btcusdt@trade",
    on_message=on_message, on_error=on_error, on_close=on_close
)
ws.on_open = on_open
ws.run_forever()
2Option 2 (async loop) → Kafka producer — Recommended
import asyncio
import json
import websockets
from confluent_kafka import Producer

# confluent-kafka has its own internal buffer — no manual buffering needed
producer = Producer({
    'bootstrap.servers': 'localhost:9092',
    'linger.ms': 5,          # wait up to 5ms to batch messages
    'batch.num.messages': 100 # or until 100 messages — whichever first
})

async def stream_to_kafka():
    url = "wss://stream.binance.com:9443/ws/btcusdt@trade"

    print("Connecting to Binance WebSocket...")

    async with websockets.connect(url) as ws:
        print("Connected. Streaming to Kafka...")

        async for message in ws:
            try:
                data = json.loads(message)

                kafka_msg = {
                    "symbol":    data["s"],
                    "price":     float(data["p"]),
                    "quantity":  float(data["q"]),
                    "trade_id":  data["t"],
                    "timestamp": data["T"]
                }

                # produce() — non-blocking, goes to confluent-kafka internal buffer
                # librdkafka batches and sends to Kafka automatically
                # No manual buffer needed — this is confluent-kafka's advantage
                producer.produce(
                    topic='crypto-prices',
                    key='BTC',
                    value=json.dumps(kafka_msg).encode('utf-8')
                )

                # poll() handles delivery callbacks — pass 0 for non-blocking check
                producer.poll(0)

            except json.JSONDecodeError as e:
                print(f"JSON parse error: {e}")

async def main():
    try:
        await stream_to_kafka()
    finally:
        # Always flush before exit — sends remaining buffered messages
        producer.flush()
        print("Flushed and closed.")

asyncio.run(main())
Why produce() then poll(0)? produce() puts the message in the internal buffer. poll(0) lets confluent-kafka process delivery callback events without blocking. Calling poll(0) after each produce() prevents the callback queue from growing unbounded in high-throughput scenarios.

confluent-kafka internal buffer — visualised

No manual buffer needed when using confluent-kafka. The linger.ms and batch.num.messages configs control how long to wait before sending a batch. Default is already sensible. Just produce() and flush().
