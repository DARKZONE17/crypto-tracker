Start Here

What is Kafka?

Before touching any code, build the mental picture first. You need to understand why Kafka exists before you can understand what it does.

The problem Kafka solves

Imagine you're building a crypto price tracker. Your Python script fetches prices every minute. Now multiple systems need that data:

Without Kafka — your script must call everyone directly:
Your Script → Database (slow, blocks) Your Script → Spark (crashes, blocks) Your Script → Dashboard (timeout, blocks) Your Script → Alert system (error, blocks)
If Spark is slow → your script waits → prices are stale. One failure = everything fails.
Kafka's solution: Your script sends data to Kafka once and moves on. Kafka holds it. Every other system reads from Kafka independently, at their own speed. None of them talk to each other or to your script. They're completely decoupled.
With Kafka — your script only talks to Kafka:
Your Script → Kafka ✓ (fast, non-blocking)
Kafka → Database (reads when ready) Kafka → Spark (reads when ready) Kafka → Dashboard (reads when ready) Kafka → Alerts (reads when ready)
Each system is independent. One crashing doesn't affect the others.

One-line definition

Kafka is a distributed log system — it receives data, stores it in order, and lets multiple systems read it at their own speed, independently.

Real-world analogy
A newspaper printing press. The press prints today's news (producer sends data). Thousands of copies are stacked and available (Kafka stores it). Millions of people buy and read it at different times (multiple consumers read independently). Reading one copy doesn't destroy the others.

Core Concepts

Every Kafka Building Block — Explained Simply

Read these in order. Each concept builds on the last. Don't skip ahead — they connect to each other.

1 · Broker

In plain English: A broker is simply a server — a computer — running the Kafka software. Nothing more. When you run Kafka on your laptop using Docker, your laptop IS the broker. It's the middleman that receives, stores, and serves all the data.

Why "broker"? Like a real-estate broker who sits between buyers and sellers, the Kafka broker sits between producers (who send data) and consumers (who read data). It holds everything in the middle and makes sure both sides can operate independently.
Zomato analogy
Zomato's backend server. When you place an order, Zomato's server receives it, stores it, and makes it visible to the delivery agent. It doesn't cook the food, doesn't eat it — it just manages the flow between the two sides.
Your laptop running Docker:
└── Kafka Broker (port 9092)
├── receives messages from producers
├── stores them on disk
└── serves them to consumers on demand

In production at companies like Swiggy or Uber: 3 or more brokers form a cluster for reliability. On your laptop for learning: 1 broker is enough.

2 · Topic

In plain English: A topic is a named category — like a folder — where you store a specific type of message. You give it a name. You create as many as you need for different data types.

Think of it exactly like a database table name. In SQL you'd have a users table and an orders table. In Kafka you'd have a crypto-prices topic and a user-events topic. Data of the same type goes to the same topic. The producer and consumer both use the same topic name to talk about the same data.
Zomato analogy
Domino's has its own order queue. McDonald's has its own. You don't mix orders from different restaurants. In Kafka: the crypto-prices topic only ever contains crypto price data. 
# Producer sends to a topic by name:
producer.send("crypto-prices", value=data)

# Consumer reads from the same topic name:
# They're connected by this shared name — that's it
consumer = KafkaConsumer("crypto-prices")
Key difference from a normal queue: messages in a Kafka topic are NOT deleted after being read. They stay for 7 days (default). Multiple systems can all read the same message. A queue would destroy the message the moment the first consumer reads it.

3 · Partition  MOST IMPORTANT CONCEPT

In plain English: Kafka splits each topic into multiple buckets called partitions. Messages are spread across these buckets. This is what allows Kafka to handle millions of messages — multiple machines can process different partitions at the same time, in parallel.

Why partitions exist — the supermarket analogy:
One checkout lane → one cashier serves everyone → slow. Five checkout lanes → five cashiers work simultaneously → fast. Partitions are those checkout lanes. Each partition is a separate, independent unit that a different consumer can handle at the same time.
Topic: "crypto-prices" split into 3 partitions:
Partition 0: [offset 0: BTC] [offset 1: BTC] [offset 2: BTC] ← only BTC here
Partition 1: [offset 0: ETH] [offset 1: ETH] ← only ETH here
Partition 2: [offset 0: SOL] [offset 1: SOL] ← only SOL here
Three consumers can read these three partitions simultaneously — 3x faster than one consumer reading one at a time.
Zomato analogy
Instead of one chef cooking everything, the kitchen is split into sections. Pizza section, Drinks section, Dessert section — all cooking at the same time. Orders get routed to the right section. Everything is done faster because it's parallel.

How does Kafka decide which partition a message goes to?

# Case 1: You provide a KEY
# Kafka hashes the key to a number → uses that to pick a partition
# Same key ALWAYS goes to the same partition → guaranteed ordering per key
producer.send("crypto-prices", key="BTC", value=data)
# "BTC" → always Partition 0
# "ETH" → always Partition 1
# "SOL" → always Partition 2

# Case 2: No key → Round-robin (messages spread evenly)
# msg1 → P0, msg2 → P1, msg3 → P2, msg4 → P0, msg5 → P1 ...
producer.send("crypto-prices", value=data)
⚠ Order is guaranteed ONLY within a single partition. Since all BTC messages go to Partition 0, BTC messages are always in sequence. But there's no ordering guarantee between different partitions (between BTC and ETH messages).

4 · Producer

In plain English: A producer is your code — the Python script you write — that connects to Kafka and sends (produces) messages into a topic. Any application that puts data into Kafka is a producer. In your project, the script that calls CoinGecko and sends prices is the producer.

The producer's job is simple: fetch data → package it → send to Kafka → move on. It doesn't wait to see who reads the data. It doesn't know anything about the consumers. It just sends and forgets.
Zomato analogy
You (the customer) place an order. You don't know which delivery agent will pick it up, which kitchen section will cook it, or how long it'll take. You just place the order and walk away. 
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    # Kafka only stores bytes, not Python dicts
    # So we convert: dict → JSON string → bytes
    value_serializer=lambda x: json.dumps(x).encode('utf-8')
)

producer.send("crypto-prices", value={"coin": "BTC", "price": 67000})
send() is asynchronous — it does NOT wait for Kafka to confirm. It puts messages into an internal buffer and sends them in batches for speed. Always call producer.flush() at the end of your script to force the buffer to empty before the program exits.
Async (non-blocking) Batches messages Auto-retries on failure

5 · Consumer

In plain English: A consumer is your code that connects to Kafka and reads (consumes) messages from a topic. It's the other side of the pipeline. Where the producer puts data in, the consumer takes data out and processes it.

The critical difference from a normal queue: When a consumer reads a message from Kafka, the message is not deleted. It stays in Kafka for the full retention period (7 days by default). This means 5 different consumers can all independently read the same message. A traditional queue destroys the message the moment the first consumer reads it.
Zomato analogy
The delivery agent picks up the order details and delivers it. But unlike a physical order that disappears, in Kafka the "order record" stays. So a second agent (like your Spark pipeline) can also pick up the same record and do something different with it — like store it in a database for analytics. 
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'crypto-prices',                  # which topic to read from
    bootstrap_servers='localhost:9092',
    group_id='my-app',                # explained in Consumer Group section
    auto_offset_reset='earliest',     # start from the beginning
    # Reverse of what producer did: bytes → string → dict
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

# This for loop runs forever — it BLOCKS and waits for messages
# When a message arrives, it processes it, then waits for the next one
for message in consumer:
    print(message.value)       # the data: {"coin": "BTC", "price": 67000}
    print(message.partition)   # which partition it came from: 0
    print(message.offset)      # its position in that partition: 47

6 · Consumer Group

In plain English: A consumer group is multiple consumer instances that share the same group_id. Kafka automatically divides the partitions between them so they share the work. This is how you read faster — instead of one consumer reading all 3 partitions, you have 3 consumers each reading 1 partition simultaneously.

The golden rule: Each partition is assigned to exactly ONE consumer within a group at a time. No two consumers in the same group ever read the same partition at the same time. Kafka handles this assignment automatically.
3 Consumers, 3 Partitions (Perfect)
Consumer A → Partition 0
Consumer B → Partition 1
Consumer C → Partition 2
✓ All partitions covered, perfectly balanced
2 Consumers, 3 Partitions
Consumer A → Partition 0 + 1
Consumer B → Partition 2
 
~ Still works, A just does more work
⚠ What if you add a 4th consumer to a group with 3 partitions? The 4th consumer sits idle — there's no partition for it to read. You can never have more active consumers than partitions in a group.

Two different group_ids = two independent readers:

# group "spark-pipeline" reads ALL messages, for Spark processing
KafkaConsumer('crypto-prices', group_id='spark-pipeline')

# group "postgres-writer" ALSO reads ALL the same messages, independently
# Both groups get a full copy of every message. They don't interfere.
KafkaConsumer('crypto-prices', group_id='postgres-writer')
Zomato analogy
3 delivery agents assigned to the same restaurant. Agent A covers the pizza kitchen (Partition 0). Agent B covers drinks (Partition 1). Agent C covers desserts (Partition 2). No two agents ever pick up from the same kitchen section. Everyone works independently, in parallel.

7 · Offset

In plain English: An offset is just a number — the position of a message inside a partition. Message 0, message 1, message 2, and so on. It's like a page number in a book. Kafka uses these numbers to know which messages each consumer group has already read, so it knows where to resume after a restart.

Partition 0 — messages stored permanently in order:
[offset 0] {"coin":"BTC","price":60000} ← read ✓
[offset 1] {"coin":"BTC","price":61000} ← read ✓
[offset 2] {"coin":"BTC","price":62000} ← consumer is HERE right now
[offset 3] {"coin":"BTC","price":63000} ← not yet read
Consumer crashes → restarts → Kafka checks "spark-pipeline was at offset 2" → resumes at offset 3. Zero data lost.
Why this matters for you: When your consumer.py crashes or you stop and restart it, it needs to know where to pick up. Kafka looks up the last committed offset for your group_id and resumes from the next message. Without offsets, you'd either re-read everything from the beginning or skip everything that arrived while you were down.
Zomato analogy
The delivery agent keeps a logbook: "Delivered orders #0, #1, #2. Next pickup is #3." Agent goes home, comes back next morning, opens logbook, sees they left at #2, picks up #3. Never re-delivers an old order, never skips one. 
# auto_offset_reset: what to do when this group_id has NO saved offset yet
# (i.e., the very FIRST time this group reads this topic)

auto_offset_reset='earliest'   # → go to offset 0, read ALL stored messages
auto_offset_reset='latest'     # → start from NOW, only read new messages

# enable_auto_commit=True saves your current offset to Kafka every second
# This is what lets you restart and resume exactly where you left off

Replication & Fault Tolerance

In plain English: Replication means Kafka automatically copies each partition onto multiple brokers (servers). If one server dies, another takes over automatically with no data loss and no downtime. This is how Kafka is "fault tolerant."

Leader and Follower: For every partition, exactly one broker is elected as the Leader — all reads and writes go through it. The other brokers are Followers — they silently copy everything the leader stores. If the leader crashes, one follower is automatically promoted to leader. Your producers and consumers don't need to do anything — it just works.
replication-factor = 3 means 3 copies of each partition:
Broker 1 (Leader for Partition 0) ← all writes go here
Broker 2 (Follower) ← silently copies Broker 1
Broker 3 (Follower) ← silently copies Broker 1
→ Broker 1 crashes → Broker 2 is elected Leader → zero interruption → zero data loss
⚠ Always use replication-factor = 3 in production. On your local laptop with 1 Docker broker, use 1. Setting it to 1 in production means if that one server crashes, all data in that partition is permanently gone.

Message Retention

In plain English: Kafka keeps messages for a fixed time window even after they've been consumed. The default is 7 days. After that, old messages are deleted to free up disk space. This is the biggest conceptual difference between Kafka and a message queue.

Why this is a superpower: Imagine you set up your Spark pipeline 3 days after the producer started sending prices. With a normal queue, those 3 days of data are gone. With Kafka, all 3 days of messages are still on disk. Your new Spark consumer can set auto_offset_reset='earliest' and replay all of it from the beginning. Time travel for data.

Delivery Semantics — The Big Three

This answers one question: "What guarantee does Kafka give me about whether a message is actually delivered?" There are 3 levels. This is one of the most common Kafka interview questions.

At-Most-Once

Sent once. If it fails — it's gone. No retry.

acks = 0

Use for: logs, metrics where losing a few is fine.

Risk: data loss possible.

At-Least-Once

Retried until confirmed. May arrive more than once.

acks = 1, retries = 3

Use for: most cases — duplicates are tolerable.

Risk: duplicates possible.

Exactly-Once

Delivered exactly once. No loss, no duplicates.

enable.idempotence = True

Use for: payments, financial transactions.

Cost: slightly slower to configure.

ZooKeeper vs KRaft — Who Manages Kafka Itself?

Kafka brokers need something to coordinate with each other — who is the leader of each partition? Which brokers are alive? That coordination layer has changed over time, and you'll see both mentioned.

ZooKeeper (old — pre-2023)

A completely separate service that Kafka depended on. You had to run TWO systems. More complexity, more things that could break.

ZooKeeper + Kafka = 2 systems

KRaft (new — Kafka 3.3+)

Kafka now manages its own coordination internally. No ZooKeeper needed. Simpler, faster, scales better.

Kafka only = 1 system ✓
If you're learning Kafka now (2025+), you're using KRaft. ZooKeeper was deprecated in Kafka 3.5. Your Docker Compose file likely already uses KRaft mode — you don't need to think about this further.

Decision Guide

When to Use Kafka — and When NOT To

Kafka is powerful but heavy. Knowing when NOT to use it is just as important as knowing when to use it. Don't use a sledgehammer to crack a nut.

✓ Use Kafka When

  • →Real-time pipelines (like your crypto project)
  • →Multiple services all need the same event
  • →You need to replay historical data
  • →Very high volume: millions of messages/sec
  • →Microservices that communicate asynchronously

✗ Don't Use Kafka When

  • →Simple task queues → use Celery / RabbitMQ
  • →Low volume (under 1000 messages/day) — overkill
  • →Request-response patterns → use REST or gRPC
  • →Small team, tight infra budget
  • →You need complex queries → use a database

2Create the topic first — run this ONCE

By default Kafka auto-creates topics with 1 partition. If you want 3 partition set num_partitions to 3.
Run this script exactly once before anything else:

# create_topic.py — run ONCE, then you're done
from kafka.admin import KafkaAdminClient, NewTopic

admin = KafkaAdminClient(bootstrap_servers='localhost:9092')

topic = NewTopic(
    name='crypto-prices',
    num_partitions=3,       # one per coin: BTC, ETH, SOL
    replication_factor=1    # 1 because you only have 1 broker on your laptop
                            # In production with 3 brokers, you'd set this to 3
)

admin.create_topics([topic])
print("Topic created!")
admin.close()
python create_topic.py
# Topic created!
⚠ Run this only once. If you run it twice, it throws a TopicAlreadyExistsException — that's fine, just ignore it. The topic was already created the first time.
3Understand every Producer parameter — line by line

Read every comment. Don't copy-paste blindly — understand what each line is doing and why:

from kafka import KafkaProducer
import json

producer = KafkaProducer(

    # bootstrap_servers: the address of your Kafka broker
    # This is the first thing your script does — connect to Kafka here
    # 'localhost:9092' = Kafka running on your machine, port 9092
    # In production you'd put the real server IP: '10.0.0.5:9092'
    bootstrap_servers='localhost:9092',

    # value_serializer: how to convert your Python dict into bytes
    # WHY: Kafka is language-agnostic — it stores raw bytes, not Python objects
    # If you don't convert, Kafka won't accept your message
    #   Step 1: json.dumps(x)    converts dict {"coin":"BTC"} → string '{"coin":"BTC"}'
    #   Step 2: .encode('utf-8') converts string → bytes b'{"coin":"BTC"}'
    value_serializer=lambda x: json.dumps(x).encode('utf-8'),

    # key_serializer: same idea for the key
    # The key is a string like "BTC" — we just encode it to bytes
    key_serializer=lambda x: x.encode('utf-8'),

    # acks: how many Kafka brokers must confirm before send() returns success
    # acks=0   → don't wait for any confirmation (fastest, risky — can lose messages)
    # acks=1   → leader broker confirms it received the message (good default)
    # acks='all'→ ALL replica brokers confirm (safest, slightly slower)
    acks=1,

    # retries: if the send fails (network hiccup, broker overloaded),
    # retry automatically this many times before giving up
    retries=3,

    # request_timeout_ms: if the broker doesn't respond in this many ms, give up
    request_timeout_ms=30000,  # 30 seconds
)
Why do we need a serializer? Imagine sending a letter internationally. You write it in Hindi. The postal system only handles sealed envelopes with standardized addresses. You must seal the letter (encode to bytes) before the postal system (Kafka) can carry it. At the destination, the recipient opens the envelope (decodes) and reads the letter (parses JSON).
4Write producer.py — the complete crypto script

This script runs in a loop forever, fetches BTC/ETH/SOL prices every 60 seconds, and sends them to Kafka with a key so each coin always goes to the same partition:

# producer.py
from kafka import KafkaProducer
import json, time, requests

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda x: json.dumps(x).encode('utf-8'),
    key_serializer=lambda x: x.encode('utf-8'),
    acks=1,
    retries=3,
)

def fetch_crypto_prices():
    # CoinGecko is free and needs no API key
    url = "https://api.coingecko.com/api/v3/simple/price"
    params = {"ids": "bitcoin,ethereum,solana", "vs_currencies": "usd"}
    return requests.get(url, params=params).json()
    # returns: {"bitcoin": {"usd": 67000}, "ethereum": {"usd": 3400}, ...}

def send_prices():
    data = fetch_crypto_prices()

    # Map CoinGecko IDs to short symbols we use as Kafka keys
    coin_map = {"bitcoin": "BTC", "ethereum": "ETH", "solana": "SOL"}

    for coin_id, symbol in coin_map.items():
        message = {
            "symbol": symbol,
            "price_usd": data[coin_id]["usd"],
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S")
        }

        # key=symbol is crucial:
        # Kafka hashes "BTC" → always maps to Partition 0
        # Kafka hashes "ETH" → always maps to Partition 1
        # Kafka hashes "SOL" → always maps to Partition 2
        # This guarantees all BTC messages are in order within Partition 0
        future = producer.send(
            topic='crypto-prices',
            key=symbol,
            value=message
        )

        # future.get() BLOCKS until Kafka confirms it received the message
        # Without this, send() is fire-and-forget — you don't know if it worked
        # meta.partition tells you which partition the message landed in
        # meta.offset tells you its position number in that partition
        meta = future.get(timeout=10)
        print(f"✓ Sent {symbol}: partition={meta.partition}, offset={meta.offset}")

    # flush() forces any remaining buffered messages to send RIGHT NOW
    # The producer batches messages for efficiency — flush() empties that batch
    # CRITICAL: without this, messages in the buffer may be lost when script exits
    producer.flush()

# Run forever. Fetch + send every 60 seconds.
while True:
    send_prices()
    time.sleep(60)
5Understand every Consumer parameter — line by line
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(

    # Which topic to read from — must match the producer's topic name exactly
    'crypto-prices',

    bootstrap_servers='localhost:9092',

    # group_id: the name of your consumer group — this is your BOOKMARK NAME
    # Kafka tracks "how far has group 'spark-pipeline' read?" separately per topic
    # When your consumer restarts, Kafka looks up this name and resumes from there
    # WITHOUT group_id: kafka-python generates a RANDOM name each restart
    #   → no saved bookmark → reads from the beginning every single time
    group_id='spark-pipeline',

    # auto_offset_reset: what to do when this group_id has NO bookmark yet
    # (only applies the very FIRST time you run with a new group_id)
    # 'earliest' → go to offset 0, read everything Kafka has stored (up to 7 days)
    # 'latest'   → start from NOW, only read new messages from this moment forward
    auto_offset_reset='earliest',

    # value_deserializer: reverse of what the producer did
    # Producer converted: dict → JSON string → bytes
    # Consumer must reverse: bytes → JSON string → dict
    #   Step 1: x.decode('utf-8') converts bytes → string
    #   Step 2: json.loads(...)   converts string → dict
    value_deserializer=lambda x: json.loads(x.decode('utf-8')),

    # key_deserializer: reverse for the key
    # 'if x else None' handles the edge case where no key was set (key is null)
    key_deserializer=lambda x: x.decode('utf-8') if x else None,

    # enable_auto_commit: automatically save your current offset to Kafka
    # True = Kafka saves "spark-pipeline was at offset 47" every second
    #        so on restart, you resume from offset 48 automatically
    enable_auto_commit=True,
    auto_commit_interval_ms=1000,  # save progress every 1 second
)

Every message you receive has these fields — all useful:

for message in consumer:
    message.value      # your data: {"symbol": "BTC", "price_usd": 67000, ...}
    message.key        # the key: "BTC" (tells you which coin)
    message.topic      # "crypto-prices" (confirms which topic)
    message.partition  # 0 (which of the 3 partitions this came from)
    message.offset     # 47 (its position number in that partition)
    message.timestamp  # when the producer sent it (unix milliseconds)

# consumer.py
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'crypto-prices',
    bootstrap_servers='localhost:9092',
    group_id='spark-pipeline',
    auto_offset_reset='earliest',
    value_deserializer=lambda x: json.loads(x.decode('utf-8')),
    key_deserializer=lambda x: x.decode('utf-8') if x else None,
    enable_auto_commit=True,
)

print("Consumer started. Waiting for messages...")

# This for loop is an infinite blocking loop
# It sits idle until a new message arrives, then enters the loop body

for message in consumer:

    print(f"[Partition {message.partition} | Offset {message.offset}]")
    print(f"  Coin:  {data['symbol']}")
    print(f"  Price: ${data['price_usd']:,}")
    print(f"  Time:  {data['timestamp']}")
    print()

8 The 3 mistakes every beginner makes — know them before you hit them
❌ Mistake 1 — Consumer re-reads everything from offset 0 every restart

Cause: You forgot group_id. Without it, kafka-python generates a brand new random group ID on every single run. Since this "new" group has never read before, it has no bookmark, so it starts from offset 0 every time.

Fix: Always add group_id='your-app-name'. Use the exact same name every run. Now Kafka stores a permanent bookmark under that name and resumes from where you left off.

❌ Mistake 2 — message.value shows bytes like b'{...}' instead of a dict

Cause: You forgot value_deserializer on the consumer. Kafka delivers raw bytes. Without the deserializer, you get the bytes directly, not the Python dict.

Fix: Add value_deserializer=lambda x: json.loads(x.decode('utf-8')). The producer serializes dict→bytes, the consumer must deserialize bytes→dict. They must mirror each other exactly.

❌ Mistake 3 — Producer script finishes but consumer received nothing

Cause: You forgot producer.flush(). The producer batches messages in an internal buffer for efficiency and sends them in bulk. If your script ends before the buffer is full enough to auto-send, those messages are silently discarded.

Fix: Always call producer.flush() before your script exits. This immediately sends everything still in the buffer to Kafka, regardless of batch size.

Final Intuition

The Mental Model — Remember This

Kafka IS

  • ✓ A distributed log system
  • ✓ A durable event store
  • ✓ A real-time data pipeline

Think like this:

Producer → writes the log

Kafka    → stores the log

Consumer → reads the log

Kafka is NOT

  • ✗ Not a queue (messages survive reads)
  • ✗ Not a database (no SQL, no queries)
  • ✗ Not a REST API (no request-response)
