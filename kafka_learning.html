What is Kafka?

Build the mental model first. You need to understand why Kafka was created before you can understand what it does.

The problem Kafka solves

You're building a crypto price tracker. Your script fetches prices every minute. Now multiple systems need that data — Spark, PostgreSQL, a live dashboard, an alert system. Without Kafka, you must call all 4 from your script. If Spark is slow, your script waits. If the DB crashes, everything stops.

Without Kafka — tight coupling:
Your Script → Database (blocks if slow) Your Script → Spark (crashes, everything stops) Your Script → Dashboard (times out, whole flow fails) Your Script → Alerts (one failure = all fail)
Kafka's solution: Your script sends to Kafka once and moves on. Kafka holds the data. Every other system reads from Kafka independently, at its own pace. None of them know or care about each other.
With Kafka — decoupled architecture:
Your Script → Kafka ✓ (non-blocking, always fast)
Kafka → Database (reads when ready, independently) Kafka → Spark (reads when ready, independently) Kafka → Dashboard (reads when ready, independently) Kafka → Alerts (reads when ready, independently)
One system crashing doesn't affect any other system.

One-line definition

Kafka is a distributed log system — it receives data, stores it in order on disk, and lets multiple systems read it independently at their own speed.

Analogy
A newspaper printing press. The press prints news (producer sends data). Copies stack up (Kafka stores messages). Thousands of people buy and read independently (multiple consumers). Reading one copy doesn't remove it for others.

Before Kafka vs After Kafka

The core architectural shift Kafka enables:

❌ Before Kafka

Application → Database
  • ✗ Tight coupling — systems depend on each other
  • ✗ No real-time data flow
  • ✗ Poor scalability
  • ✗ Data loss if one system fails

✓ After Kafka

Producer → Kafka → Consumer
  • ✓ Decoupled — systems are independent
  • ✓ Real-time streaming
  • ✓ Horizontally scalable
  • ✓ Fault-tolerant via replication

Core Concepts

Every Kafka Building Block

Read these in order. Each concept builds on the last. Don't skip ahead.

1 · Broker

In plain English: A broker is simply a computer running Kafka software. When you run Kafka on your laptop with Docker, your laptop IS the broker. It sits in the middle — receiving data from producers and serving it to consumers.

Why "broker"? Like a real-estate broker between buyer and seller, a Kafka broker sits between producers and consumers. It holds everything in the middle so both sides are independent.
Zomato analogy
Zomato's backend server. Receives your order, stores it, serves it to the delivery agent. Doesn't place orders, doesn't deliver — just manages the flow.
Your laptop running Docker: └── Kafka Broker (port 9092) ├── receives messages from producers ├── stores them on disk └── serves them to consumers on demand

In production: 3+ brokers form a cluster. Locally for learning: 1 is enough.

2 · Topic

In plain English: A topic is a named category — like a folder — for a specific type of message. You name it. You create as many as you need.

Exact same idea as a database table name. In SQL: users table and orders table. In Kafka: hello-topic and crypto-prices topic. Same type of data → same topic.
Zomato analogy
Domino's has its own order queue. McDonald's has its own. You never mix them. In Kafka: crypto-prices only ever has crypto price data.
# Both producer and consumer use the same topic name to find each other
producer.send("hello-topic", value=data)
consumer = KafkaConsumer("hello-topic")
Key difference from a normal queue: messages are NOT deleted after being read. They stay for 7 days by default. Multiple consumers can all read the same message independently.

3 · Partition  MOST IMPORTANT

In plain English: Kafka splits each topic into multiple buckets called partitions. Messages spread across them. Multiple consumers can read different partitions simultaneously — this is how Kafka handles millions of messages.

Supermarket checkout analogy: One cashier lane = slow. Five lanes = 5 cashiers simultaneously = fast. Partitions are those lanes. Each is processed independently, in parallel.
Topic: "crypto-prices" split into 3 partitions: Partition 0: [offset 0: BTC] [offset 1: BTC] [offset 2: BTC] ← BTC only Partition 1: [offset 0: ETH] [offset 1: ETH] ← ETH only Partition 2: [offset 0: SOL] [offset 1: SOL] ← SOL only Three consumers can read these simultaneously → 3× faster than one consumer alone.
Zomato analogy
Pizza section, Drinks section, Dessert section — all cooking simultaneously. Orders routed to the right section. Everything done faster because it's parallel.
# With KEY → same key always → same partition (guaranteed ordering per key)
producer.send("crypto-prices", key="BTC", value=data)
# "BTC" → Partition 0 always, "ETH" → Partition 1, "SOL" → Partition 2

# No KEY → Round-robin (msg1→P0, msg2→P1, msg3→P2, msg4→P0 ...)
producer.send("crypto-prices", value=data)
⚠ Order is guaranteed ONLY within one partition. All BTC messages are in order in P0. But no ordering guarantee exists across different partitions.

4 · Producer

In plain English: A producer is your Python script that connects to Kafka and sends messages into a topic. It doesn't know who will read the data. Doesn't care. Just sends and moves on.

The flow: Fetch prices from CoinGecko → package as JSON → convert to bytes (Kafka only stores bytes) → send to Kafka topic. Done. Your script moves to the next price.
Zomato analogy
You (the customer) place an order. You don't know which agent will deliver, which kitchen section cooks it. You just place the order and walk away.
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    # Kafka stores bytes — convert dict → JSON string → bytes
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
producer.send('hello-topic', value={"message": "Hello World"})
send() is async — it doesn't wait for confirmation. Messages buffer internally and send in batches. Always call producer.flush() before your script exits or buffered messages are silently discarded.
Async (non-blocking) Batches for speed Auto-retries

5 · Consumer

In plain English: A consumer is your Python script that connects to Kafka and reads messages from a topic. It's the other side of the pipeline. It runs forever in a loop, waiting for messages to arrive.

Critical difference from a normal queue: Reading a message does NOT delete it. It stays in Kafka for the full retention period (7 days). Five different consumers can all read the same message independently. A traditional queue destroys messages the moment the first consumer reads them.
Zomato analogy
The delivery agent. They pick up order details. But unlike a physical order, the Kafka "record" stays — so a second agent (like your Spark pipeline) can also process it independently.
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'hello-topic',
    bootstrap_servers='localhost:9092',
    auto_offset_reset='earliest',
    group_id='my-group',
    # Reverse of producer: bytes → string → dict
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

for message in consumer:          # runs forever, waiting for messages
    print("Received:", message.value)  # already a dict

6 · Consumer Group

In plain English: Multiple consumers that share the same group_id. Kafka divides partitions between them automatically. More consumers = faster reading. This is load balancing built into Kafka.

Golden rule: Each partition → exactly one consumer per group at a time. With 3 partitions and 3 consumers, each handles 1 partition. Can't have more active consumers than partitions.
3 consumers, 3 partitions (ideal)
Consumer A → Partition 0 Consumer B → Partition 1 Consumer C → Partition 2
✓ Perfect balance
2 consumers, 3 partitions
Consumer A → Partition 0 + 1 Consumer B → Partition 2
~ A does more work
⚠ 4th consumer with 3 partitions = sits idle. Never more active consumers than partitions in a group.
# Different group_ids = fully independent readers, each gets ALL messages
KafkaConsumer('hello-topic', group_id='my-group')
KafkaConsumer('hello-topic', group_id='spark-pipeline')  # also reads everything

7 · Offset

In plain English: An offset is a number — the position of a message inside a partition. Message 0, 1, 2… Like a page number. Kafka uses offsets to know where each consumer group left off, so restarts resume exactly from where they stopped.

Partition 0 — messages stored permanently in order: [offset 0] {"message":"Hello World 1"} ← read ✓ [offset 1] {"message":"Hello World 2"} ← read ✓ [offset 2] {"message":"Hello World 3"} ← consumer is HERE [offset 3] {"message":"Hello World 4"} ← not yet read Consumer crashes → restarts → Kafka sees "my-group was at offset 2" → resumes at 3. Zero loss.
Zomato analogy
The delivery agent's logbook: "Delivered orders #0, #1, #2. Next is #3." Goes home, comes back, opens logbook, resumes at #3. Never re-delivers old orders, never skips.
# auto_offset_reset: where to start when this group_id has no saved offset yet
# (only applies the very FIRST time you run with a new group_id)
auto_offset_reset='earliest'  # → start from offset 0, read everything stored
auto_offset_reset='latest'    # → start from NOW, only new messages going forward

Replication & Fault Tolerance

In plain English: Kafka automatically copies each partition onto multiple brokers. If one server dies, another takes over instantly with zero data loss.

Leader + Follower: One broker is the Leader per partition — all reads/writes go through it. Others are Followers — they silently mirror. Leader crashes → Follower auto-promoted → zero downtime.
replication-factor = 3 (3 copies of each partition): Broker 1 (Leader) ← all writes go here Broker 2 (Follower) ← silently mirrors Broker 1 Broker 3 (Follower) ← silently mirrors Broker 1 Broker 1 crashes → Broker 2 becomes Leader automatically
⚠ Use replication-factor = 3 in production. On laptop with 1 Docker broker, use 1. replication-factor=1 in production = data loss on server crash.

Message Retention

In plain English: Kafka keeps messages for 7 days (default) even after they've been consumed. After that, old messages are deleted. This is what makes Kafka a log, not a queue.

Why this is powerful: Add a new Spark pipeline 3 days after the producer started. Set auto_offset_reset='earliest' and it replays all 3 days of data. A normal queue would have deleted those messages the moment the first consumer read them.
Analogy
A DVR recording a show. Airs live (producer streams). Stored on disk (Kafka holds it). Watch any time within the window (consumers replay). One person watching doesn't delete it for others.

Delivery Semantics — The Big Three

What guarantee does Kafka give about whether a message is actually delivered? Three levels — a classic interview question.

At-Most-Once

Sent once. Fails → gone. No retry.

acks = 0

Use: logs, metrics. Risk: data loss.

At-Least-Once

Retried until confirmed. May arrive more than once.

acks = 1, retries = 3

Use: most cases. Risk: duplicates.

Exactly-Once

Once exactly. No loss, no duplicates.

enable.idempotence = True

Use: payments, finance. Cost: slower.

ZooKeeper vs KRaft

Kafka brokers need something to coordinate: who is partition leader? Which brokers are alive? That coordination layer changed.

ZooKeeper (old)

Separate service. You ran TWO systems. More complexity, more failure points. Your docker-compose.yml includes ZooKeeper — that's this.

ZooKeeper + Kafka = 2 systems

KRaft (new — Kafka 3.3+)

Kafka manages its own metadata internally. No ZooKeeper needed. Simpler, faster, scales better. Future of Kafka.

Kafka only = 1 system ✓
Your current docker-compose.yml uses ZooKeeper (the Confluent 7.4.0 image). That's perfectly fine for learning. Just know that in newer setups, ZooKeeper is replaced by KRaft.

Decision Guide

When to Use Kafka — and When NOT To

✓ Use Kafka When

  • →Real-time data pipelines
  • →Multiple services need the same event
  • →You need to replay historical data
  • →Very high volume: millions of messages/sec
  • →Microservices communicating asynchronously

✗ Don't Use Kafka When

  • →Simple tasks → use Celery / RabbitMQ
  • →Low volume (< 1000 messages/day) — overkill
  • →Request-response → use REST or gRPC
  • →Small team, tight infra budget
  • →Complex queries needed → use a database

Docker Setup — Why Docker & How to Use It

Before writing any Python, you need Kafka running. Docker is the simplest way to do this on your laptop without a complex manual install.

Why Docker?

Without Docker

  • ✗ Manual Kafka + ZooKeeper installation
  • ✗ Complex config files
  • ✗ Version conflicts
  • ✗ Hard to debug

With Docker

  • ✓ One command setup
  • ✓ Clean, isolated environment
  • ✓ Reproducible on any machine
  • ✓ Easy to tear down and restart

Step 1 · Create docker-compose.yml

Create this file in your project folder. It defines two containers: ZooKeeper (metadata coordinator) and Kafka (the broker).

services:

  zookeeper:
    image: confluentinc/cp-zookeeper:7.4.0
    # ZooKeeper manages Kafka metadata — who is leader, which brokers are alive
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181   # port ZooKeeper listens on

  kafka:
    image: confluentinc/cp-kafka:7.4.0
    depends_on:
      - zookeeper             # Kafka must start after ZooKeeper
    ports:
      - "9092:9092"           # expose port 9092 so your Python scripts can connect
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181   # tell Kafka where ZooKeeper is
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1   # 1 because we have 1 broker

Step 2 · Start and Stop Kafka

Terminal — start Kafka (run once, keep running)
docker-compose up -d # -d = detached, runs in background so it doesn't block your terminal
Verify both containers are up
docker ps # CONTAINER ID IMAGE STATUS # abc123 confluentinc/cp-kafka:7.4.0 Up 2 minutes # def456 confluentinc/cp-zookeeper:7.4.0 Up 2 minutes
Stop Kafka when done
docker-compose down
Kafka is now running at localhost:9092. That's the address your Python scripts will use to connect.

Step 3 · Create a Topic via CLI

Create a topic
docker exec -it kafka-project-1-kafka-1 kafka-topics \ --create \ --topic hello-topic \ --bootstrap-server localhost:9092 \ --partitions 1 \ --replication-factor 1 # Created topic hello-topic.
Verify the topic exists
docker exec -it kafka-project-1-kafka-1 kafka-topics \ --list \ --bootstrap-server localhost:9092 # hello-topic # __consumer_offsets ← internal Kafka topic, ignore this
Breaking down the command:
kafka-project-1-kafka-1 = the Docker container name (check yours with docker ps)
kafka-topics --create = the Kafka CLI tool to create a topic
--partitions 1 = start with 1 partition (enough for hello world)
--replication-factor 1 = 1 copy since we have 1 broker locally
Choose Your Client

kafka-python vs confluent-kafka

There are two popular Python libraries for talking to Kafka. They both connect to the same Kafka broker — but their APIs are completely different. You cannot mix them. Pick one and stay consistent.

kafka-python

pip install kafka-python

Pure Python. Easier to learn. More beginner-friendly API. Perfect for learning Kafka concepts and personal/study projects.

Use when

  • →Learning Kafka for the first time
  • →Personal projects & prototyping
  • →Simpler API is a priority
  • →No Mac/librdkafka hassle needed
confluent-kafka

pip install confluent-kafka

C-extension backed by librdkafka. Faster, battle-tested, used in production at scale. Officially supported by Confluent (Kafka's creators).

Use when

  • →Production workloads & high throughput
  • →Company / work projects
  • →Confluent Cloud integration
  • →Max performance & lower latency

Installation

C-backed High throughput Production-grade
2producer.py — confluent-kafka style

Notice what's different from kafka-python: config is a dict, method is produce() not send(), and you handle serialization yourself.

# producer.py  (confluent-kafka)
from confluent_kafka import Producer
import json, time

# Config is a DICT with dot-notation keys (not keyword args like kafka-python)
producer = Producer({
    'bootstrap.servers': 'localhost:9092'
})

# Optional: delivery callback — called when broker confirms receipt
def delivery_report(err, msg):
    if err is not None:
        print(f"Delivery failed: {err}")
    else:
        print(f"✓ Delivered to partition {msg.partition()} @ offset {msg.offset()}")

i = 1
while True:
    data = {"message": f"Hello World {i}", "number": i}

    # produce() — not send()!
    # Must manually encode to bytes — no value_serializer here
    producer.produce(
        topic='hello-topic',
        value=json.dumps(data).encode('utf-8'),
        on_delivery=delivery_report   # optional callback
    )

    # flush() is REQUIRED — sends all buffered messages immediately
    producer.flush()
    print("Sent:", data)

    i += 1
    time.sleep(2)
Expected output
# ✓ Delivered to partition 0 @ offset 0 # Sent: {'message': 'Hello World 1', 'number': 1} # ✓ Delivered to partition 0 @ offset 1 # Sent: {'message': 'Hello World 2', 'number': 2}
⚠ flush() is not optional. confluent-kafka's produce() always buffers messages internally. If you skip flush(), messages may never actually leave your script. Always call flush() — either after every message or at the end of a batch.
3consumer.py — confluent-kafka style

The consumer API is more explicit — you subscribe to topics separately, use poll() in a loop, and access message fields as method calls.

# consumer.py  (confluent-kafka)
from confluent_kafka import Consumer
import json

# Config dict — note dot-notation keys, NOT underscores
consumer = Consumer({
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'my-group',           # note: 'group.id' not group_id
    'auto.offset.reset': 'earliest'   # note: 'auto.offset.reset' not auto_offset_reset
})

# subscribe() is a separate step — unlike kafka-python where you pass topic at creation
consumer.subscribe(['hello-topic'])

print("Waiting for messages...\n")

try:
    while True:
        # poll() — blocks for up to 1.0 second waiting for a message
        # Returns None if no message arrived in that window
        msg = consumer.poll(timeout=1.0)

        if msg is None:
            continue                    # no message this second, keep waiting

        if msg.error():
            print(f"Error: {msg.error()}")
            continue

        # IMPORTANT: msg.value() is a METHOD CALL — not a property!
        # Also returns bytes — must decode manually
        data = json.loads(msg.value().decode('utf-8'))

        print(f"[P{msg.partition()} | offset {msg.offset()}]")
        print(f"  Received: {data}")

except KeyboardInterrupt:
    print("\nStopping consumer...")
finally:
    consumer.close()   # always close — commits offset, releases partition
Expected output
# Waiting for messages... # # [P0 | offset 0] # Received: {'message': 'Hello World 1', 'number': 1} # [P0 | offset 1] # Received: {'message': 'Hello World 2', 'number': 2}
Key differences from kafka-python consumer:
• consumer.subscribe(['topic']) — separate step, takes a list
• consumer.poll(1.0) — explicit poll loop instead of for message in consumer
• msg.value() — method call with parentheses, returns bytes
• msg.error() — must check for errors yourself
• consumer.close() — explicitly close in a finally block
Always wrap in try/finally and call consumer.close(). This commits your current offset to Kafka and releases partition ownership cleanly. Without it, Kafka may wait for a timeout before reassigning partitions to other consumers.

from kafka.admin import KafkaAdminClient, NewTopic

admin = KafkaAdminClient(bootstrap_servers='localhost:9092')

    name='crypto-prices',
    num_partitions=3,
    replication_factor=1
)

print("Topic 'crypto-prices' created with 3 partitions!")
admin.close()
⚠ Run create_topic once only. Running it a second time throws TopicAlreadyExistsException — just ignore that error, the topic is already there.
2producer.py — Fetch and stream crypto prices
# producer.py  (kafka-python)
from kafka import KafkaProducer
import json, time, requests

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda x: json.dumps(x).encode('utf-8'),
    key_serializer=lambda x: x.encode('utf-8'),
    acks=1,
    retries=3,
)

def fetch_prices():
    url = "https://api.coingecko.com/api/v3/simple/price"
    params = {"ids": "bitcoin,ethereum,solana", "vs_currencies": "usd"}
    return requests.get(url, params=params).json()

def send_prices():
    data = fetch_prices()
    coin_map = {"bitcoin": "BTC", "ethereum": "ETH", "solana": "SOL"}

    for coin_id, symbol in coin_map.items():
        message = {
            "symbol": symbol,
            "price_usd": data[coin_id]["usd"],
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S")
        }
        future = producer.send(topic='crypto-prices', key=symbol, value=message)
        meta = future.get(timeout=10)
        print(f"✓ {symbol}: partition={meta.partition}, offset={meta.offset}")

    producer.flush()

while True:
    send_prices()
    time.sleep(60)
3consumer.py — Read and process crypto prices
# consumer.py  (kafka-python)
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'crypto-prices',
    bootstrap_servers='localhost:9092',
    group_id='spark-pipeline',
    auto_offset_reset='earliest',
    value_deserializer=lambda x: json.loads(x.decode('utf-8')),
    key_deserializer=lambda x: x.decode('utf-8') if x else None,
    enable_auto_commit=True,
    auto_commit_interval_ms=1000,
)

for message in consumer:
    data = message.value
    print(f"[P{message.partition} | offset {message.offset}]")
    print(f"  {data['symbol']}: ${data['price_usd']:,}  @ {data['timestamp']}")
Troubleshooting

Common Errors — What They Mean & How to Fix Them

Errors you will hit. Know the root cause so you fix them in seconds instead of spending an hour Googling.

Error 1 — ModuleNotFoundError: No module named 'kafka' or 'confluent_kafka'

Error you see:
ModuleNotFoundError: No module named 'kafka'
ModuleNotFoundError: No module named 'confluent_kafka'
Root cause: Python environment mismatch. You installed the library into Python A but your script is running with Python B.
Fix — use the same Python explicitly:
/usr/bin/python3 -m pip install kafka-python (or confluent-kafka)
/usr/bin/python3 producer.py 
# Cleanest fix — use a virtual environment:
python3 -m venv venv
source venv/bin/activate    # Mac/Linux
pip install kafka-python     # or: pip install confluent-kafka
python producer.py
The cleanest solution: Always use a virtual environment. pip and python always refer to the same thing within a venv.

Error 2 — ImportError on confluent_kafka (Mac)

Error you see:
ImportError: librdkafka.so.1: cannot open shared object file
or build errors when running pip install confluent-kafka
Root cause: confluent-kafka links against the C library librdkafka. On Mac, this C library may not be present, so pip fails to compile the extension.
Fix — install librdkafka first, then pip:
brew install librdkafka
pip install confluent-kafka

Error 3 — Consumer re-reads from offset 0 every restart

Symptom: Every time you restart consumer.py, it prints all messages from the beginning instead of resuming where it left off.
Root cause: Missing group_id. Without it, a random UUID is generated on every run. Since this "new" group has never read before, Kafka applies auto_offset_reset — reading from earliest every time.
Fix: Add a fixed group_id and use the same name every run.
# kafka-python — WRONG (no group_id):
consumer = KafkaConsumer('hello-topic', bootstrap_servers='localhost:9092')

# kafka-python — CORRECT:
consumer = KafkaConsumer('hello-topic', bootstrap_servers='localhost:9092',
                         group_id='my-group')

# confluent-kafka — WRONG (no group.id):
consumer = Consumer({'bootstrap.servers': 'localhost:9092'})

# confluent-kafka — CORRECT:
consumer = Consumer({
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'my-group'
})

Error 4 — message.value shows b'{...}' bytes instead of a dict

Symptom: Consumer prints Received: b'{"message": "Hello World 1"}' — note the b'...' prefix.
Root cause: Missing deserializer. Kafka delivers raw bytes. Without conversion, you get bytes directly.
Fix: Add the deserializer. Must be the reverse of what the producer's serializer did.
# kafka-python — add value_deserializer at consumer creation:
value_deserializer=lambda x: json.loads(x.decode('utf-8'))

# confluent-kafka — decode manually after poll():
data = json.loads(msg.value().decode('utf-8'))

Error 5 — Producer exits but consumer received nothing

Symptom: Producer prints "Sent: ..." but the consumer receives no messages.
Root cause: Missing flush(). Both libraries buffer messages internally. If your script exits before flush, those buffered messages are silently discarded.
Fix: Always call flush() before your script exits.
# kafka-python:
producer.send('hello-topic', value=data)
producer.flush()   # ← required

# confluent-kafka:
producer.produce('hello-topic', value=data.encode())
producer.flush()   # ← required (even more critical here)

Error 6 — AttributeError: 'Message' has no attribute 'value' (confluent-kafka)

Error you see:
AttributeError: 'Message' object has no attribute 'value'
Root cause: Using kafka-python style property access on a confluent-kafka Message object. In confluent-kafka, message fields are methods, not properties.
Fix: Add parentheses — they are method calls.
# WRONG (kafka-python style — doesn't work in confluent-kafka):
data = msg.value
part = msg.partition
off  = msg.offset

# CORRECT (confluent-kafka — all method calls):
data = msg.value()
part = msg.partition()
off  = msg.offset()

Final Intuition

Remember This

Kafka IS

  • ✓ A distributed log system
  • ✓ A durable event store
  • ✓ A real-time data pipeline
  • ✓ A distributed streaming platform

Mental model:

Producer → writes the log

Kafka    → stores the log

Consumer → reads the log

Kafka is NOT

  • ✗ Not a queue (messages survive reads)
  • ✗ Not a database (no SQL, no queries)
  • ✗ Not a processing engine (no compute)
  • ✗ Not a REST API (no request-response)

Client library choice:

kafka-python   → learning & prototyping

confluent-kafka → production & your project
