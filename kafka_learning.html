Core Concepts

Deep + Simple

Broker

Definition A broker is a Kafka server that stores and serves data.
Simple It's just the machine running Kafka.
🏢 Zomato backend server handling orders

Topic

Definition A topic is a logical channel where messages are stored.
Simple Like a database table or queue. 
producer.send("crypto-prices", value=data)
🍕 pizza-orders    🥤 drink-orders

Partition  MOST IMPORTANT

Definition A topic is split into partitions for parallel processing.
Why? To scale Kafka horizontally.
Topic: crypto-prices
Partition 0: [msg1, msg2, msg3]
Partition 1: [msg4, msg5]
Partition 2: [msg6, msg7]
Analogy Instead of ONE chef — Kitchen Sections: Pizza section, Drinks section, Dessert section. Orders distributed → faster processing.

⚠ Order is guaranteed ONLY within a partition.

Partitioning Logic

# Case 1: With Key — same key → same partition
producer.send("topic", key="BTC", value=data)

# Case 2: No Key — Round-robin distribution

Producer

Definition A producer sends data to Kafka.
Flow API → Python Producer → Kafka Topic 
from kafka import KafkaProducer
import json

producer = KafkaProducer(
   bootstrap_servers='localhost:9092',

producer.send("crypto-prices", {"coin": "BTC", "price": 50000})
📱 Customer placing order on Zomato
Async (fast) Can batch messages Can retry on failure

Consumer

Definition A consumer reads data from Kafka.
Flow Kafka → Consumer → Processing (Spark / DB) 
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
   "crypto-prices",
   bootstrap_servers='localhost:9092',
   value_deserializer=lambda x: json.loads(x.decode('utf-8')),
   auto_offset_reset='earliest'
)

for msg in consumer:
   print(msg.value)
🛵 Delivery agent picking up orders

Consumer Group  LOAD BALANCING MAGIC

Definition A group of consumers working together.

Topic has 3 partitions

Case 1: 3 Consumers
Consumer A → Partition 0
Consumer B → Partition 1
Consumer C → Partition 2
✓ Perfect load balancing
Case 2: 2 Consumers
Consumer A → Partition 0, 1
Consumer B → Partition 2
🚴 3 delivery boys, 3 kitchens → each handles one → fast delivery

⚠ One partition = one consumer at a time.

Offset  IMPORTANT BUT SIMPLE

Definition Offset = position of message in partition.
Partition 0:
[0] msg1
[1] msg2
[2] msg3
Why? To track what is already read.
📦 "I delivered order #15, next is #16"

Replication & Fault Tolerance

Replication Factor

What Each partition is copied across multiple brokers. If one broker dies, another takes over — zero data loss.
Leader One broker is the Leader for each partition — all reads/writes go through it.
Follower Other brokers silently mirror the data. If the leader crashes, a follower is automatically promoted.
replication-factor = 3
Broker 1 (Leader) ← writes here
Broker 2 (Follower) ← mirrors Broker 1
Broker 3 (Follower) ← mirrors Broker 1
→ Broker 1 crashes → Broker 2 becomes Leader automatically
🔁 Zomato keeps order records on 3 servers — if one crashes, the others still know your order

Rule of thumb: replication-factor = 3 in production. Never set it to 1 — that's a single point of failure.

Delivery Semantics — The Big Three

This is one of the most asked interview topics on Kafka. It defines what guarantee Kafka gives you about whether a message is delivered.

At-Most-Once

Message is sent once. If it fails — it's gone. No retry.

acks = 0

Use when: speed > correctness (metrics, logs)

Risk: data loss possible

At-Least-Once

Message is retried until acknowledged. May be delivered more than once.

acks = 1 or all, retries > 0

Use when: duplicates are tolerable (most cases)

Risk: duplicates possible

Exactly-Once

Message delivered exactly once. No loss. No duplicates. The gold standard.

enable.idempotence = True
transactional.id = "t1"

Use when: payments, financial data, orders

Cost: slightly slower, more config

ZooKeeper vs KRaft — The Brain of Kafka

ZooKeeper (Old Way)

What A separate service that Kafka used to manage metadata — which broker is leader, who's alive, topic configs, etc.
Problem Extra complexity. You had to run and maintain TWO systems: Kafka + ZooKeeper. More things to break.
ZooKeeper + Kafka Brokers = two systems to manage

KRaft (New Way — Kafka 3.x+)

What Kafka now manages its own metadata internally using a built-in consensus protocol. No ZooKeeper needed.
Why better Simpler deployment, faster startup, scales to millions of partitions. This is the standard going forward.
Kafka (KRaft mode) = one system. Done.
Default in Kafka 3.3+ ZooKeeper deprecated in 3.5+

Message Retention — Kafka is Not a Queue

This is what makes Kafka fundamentally different from RabbitMQ or a normal queue. Messages are NOT deleted after being consumed. They stay for a configured time window.

Default Retention

7 days

Messages kept for 7 days regardless of consumption

Config option

retention.ms = 604800000
retention.bytes = -1

Power of this

New consumers can replay ALL past messages from offset 0 — like rewinding time.

# Example: read from the very beginning
auto_offset_reset = 'earliest'   # replays all retained messages
auto_offset_reset = 'latest'    # only new messages from now
📼 Like a DVR — the show is broadcast live, but you can rewind and replay any part within the retention window

When to Use Kafka — and When NOT To

✓ Use Kafka When

  • You need real-time data pipelines (like your crypto project)
  • Multiple systems need the same event (fan-out)
  • You need event replay / time-travel
  • You process millions of messages/second
  • You need decoupling between producer and consumer
  • Log aggregation across microservices

✗ Don't Use Kafka When

  • You need simple task queues → use Celery / RabbitMQ
  • Low message volume (< 1000/day) → overkill
  • You need request-response patterns → use REST / gRPC
  • Your team is small and infra cost is a concern
  • You need complex query patterns → use a DB

Step 2 — The Producer, fully explained

Every parameter you'll actually use — with the why for each:

from kafka import KafkaProducer
import json

producer = KafkaProducer(
    # WHERE to connect — address of your broker
    # for local docker: 'localhost:9092'
    # for multiple brokers: ['host1:9092', 'host2:9092']
    bootstrap_servers='localhost:9092',

    # HOW to encode your message VALUE before sending
    # Kafka only stores bytes — you must convert dict → bytes
    # json.dumps(x) converts dict to JSON string
    # .encode('utf-8') converts string to bytes
    value_serializer=lambda x: json.dumps(x).encode('utf-8'),

    # HOW to encode your message KEY before sending
    # Key is used for partition routing — same key = same partition
    key_serializer=lambda x: x.encode('utf-8'),

    # RELIABILITY: how many brokers must confirm before success
    # acks=0    → fire and forget, fastest, can lose messages
    # acks=1    → leader broker confirms (default, good for your project)
    # acks='all'→ all replicas confirm, slowest, never loses
    acks=1,

    retries=3,                  # how many times to retry if send fails
    request_timeout_ms=30000,   # max wait for broker response (ms)
)

Sending a message — minimal vs full:

# Minimal send — no key, goes to random partition
producer.send('crypto-prices', value={"coin": "BTC", "price": 67000})

# With key — BTC always → same partition (RECOMMENDED)
producer.send(
    topic='crypto-prices',
    key='BTC',
    value={"coin": "BTC", "price": 67000, "timestamp": "2024-01-01T10:00:00"}
)

# IMPORTANT: flush() waits until all buffered messages are actually sent
# Without flush(), your script may exit before messages leave the buffer
producer.flush()
producer.close()

💡 send() is async — it doesn't block. The message sits in an internal buffer and gets sent in batches. flush() forces everything out immediately.

Step 3 — producer.py (your actual crypto script)

# producer.py
from kafka import KafkaProducer
import json, time, requests

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda x: json.dumps(x).encode('utf-8'),
    key_serializer=lambda x: x.encode('utf-8'),
    acks=1,
    retries=3,
)

def fetch_crypto_prices():
    """Fetch BTC, ETH, SOL prices from CoinGecko (free, no API key)"""
    url = "https://api.coingecko.com/api/v3/simple/price"
    params = {"ids": "bitcoin,ethereum,solana", "vs_currencies": "usd"}
    return requests.get(url, params=params).json()
    # returns: {"bitcoin": {"usd": 67000}, "ethereum": {"usd": 3400}, ...}

def send_prices():
    data = fetch_crypto_prices()
    coin_map = {"bitcoin": "BTC", "ethereum": "ETH", "solana": "SOL"}

    for coin_id, symbol in coin_map.items():
        message = {
            "symbol": symbol,
            "price_usd": data[coin_id]["usd"],
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S")
        }

        # BTC → Partition 0, ETH → P1, SOL → P2  (key ensures ordering)
        future = producer.send(topic='crypto-prices', key=symbol, value=message)

        # Wait for confirmation and print result
        meta = future.get(timeout=10)
        print(f"Sent {symbol}: partition={meta.partition}, offset={meta.offset}")

    producer.flush()

while True:
    send_prices()
    time.sleep(60)

Step 4 — The Consumer, fully explained

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    # WHICH TOPIC(S) to read from (can pass multiple)
    'crypto-prices',

    bootstrap_servers='localhost:9092',

    # YOUR GROUP NAME — Kafka uses this to track your saved offset
    # Same group_id = consumers share the work (load balance)
    # Different group_id = independent reader, sees ALL messages
    group_id='spark-pipeline',

    # WHAT TO DO if this group has never read before (no saved offset)
    # 'earliest' → start from message 0, read everything in retention
    # 'latest'   → start from NOW, only new messages going forward
    auto_offset_reset='earliest',

    # HOW TO DECODE bytes → Python object (reverse of producer)
    value_deserializer=lambda x: json.loads(x.decode('utf-8')),
    key_deserializer=lambda x: x.decode('utf-8') if x else None,

    # AUTO-SAVE offset to Kafka after reading
    # True = Kafka remembers where you left off on restart
    enable_auto_commit=True,
    auto_commit_interval_ms=1000,   # save offset every 1 second

    max_poll_records=500,           # max messages per poll() per partition
)

Every field on the message object you receive:

for message in consumer:
    print(message.value)      # {"symbol": "BTC", "price_usd": 67000, ...}
    print(message.key)        # "BTC"
    print(message.topic)      # "crypto-prices"
    print(message.partition)  # 0  (which partition this came from)
    print(message.offset)     # 47 (position in that partition)
    print(message.timestamp)  # unix timestamp in milliseconds

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'crypto-prices',
    bootstrap_servers='localhost:9092',
    group_id='spark-pipeline',
    auto_offset_reset='earliest',
    value_deserializer=lambda x: json.loads(x.decode('utf-8')),
    key_deserializer=lambda x: x.decode('utf-8') if x else None,
    enable_auto_commit=True,
)

print("Consumer started. Waiting for messages...")

for message in consumer:
    data = message.value
    print(f"[Partition {message.partition} | Offset {message.offset}]")
    print(f"  Coin:  {data['symbol']}")
    print(f"  Price: ${data['price_usd']:,}")
    print(f"  Time:  {data['timestamp']}")
    print()

Step 6 — Create the topic (run this ONCE first)

By default Kafka auto-creates topics with 1 partition. For your project you want 3 — one per coin:

# create_topic.py — run this ONCE before anything else
from kafka.admin import KafkaAdminClient, NewTopic

admin = KafkaAdminClient(bootstrap_servers='localhost:9092')

topic = NewTopic(
    name='crypto-prices',
    num_partitions=3,       # one per coin: BTC, ETH, SOL
    replication_factor=1    # 1 because you only have 1 broker locally
)

admin.create_topics([topic])
print("Topic created!")
admin.close()

Step 8 — The 3 things that trip everyone up

Know these before you hit them.

❌ "My consumer reads everything from scratch every restart"

You forgot group_id. Without it, kafka-python generates a random group ID every run — no saved offset. Add group_id='spark-pipeline' and Kafka remembers exactly where you left off.

❌ "message.value is bytes, not dict"

You forgot value_deserializer on the consumer. The serializer on the producer converts dict→bytes; the deserializer on the consumer converts bytes→dict. They must mirror each other exactly.

❌ "Producer script ended but consumer got nothing"

You forgot producer.flush(). The producer buffers messages and sends in batches. If your script exits before flushing, the buffer is discarded. Always call flush() before your script ends.
